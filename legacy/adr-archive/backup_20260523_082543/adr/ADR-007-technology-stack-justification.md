# ADR-007: Обоснование технологического стека портфолио

## Статус
Принято

## Контекст
Портфолио системного архитектора должно демонстрировать не только умение писать код, но и способность выбирать подходящие технологии для решения сложных enterprise-задач. Необходимо обосновать выбор сложного технологического стека (Kubernetes, микросервисы, полный мониторинг, mixed scripting), чтобы:
1. Показать, что сложность является осознанным выбором, а не случайным нагромождением
2. Демонстрировать экспертизу в production-grade архитектуре
3. Ответить на потенциальную критику "оверэнжиниринга" со стороны ревьюеров гранта или работодателей

## Рассмотренные варианты

### 1. Упрощенный стек (минималистичный подход)
- **Описание**: Простой монолит на Flask/FastAPI, база данных SQLite, деплой через Docker Compose
- **Преимущества**: Быстрый старт, низкий порог входа, проще поддерживать
- **Недостатки**: Не демонстрирует enterprise-навыки, ограниченная демонстрация экспертизы

### 2. Современный enterprise-стек (выбранный вариант)
- **Описание**: Kubernetes, 8 микросервисов, Prometheus/Grafana/Alertmanager, mixed scripting (PowerShell/Bash/Python), полный CI/CD
- **Преимущества**: Демонстрирует полный спектр архитектурных навыков, релевантен для российского enterprise-рынка
- **Недостатки**: Высокая сложность, требует больше времени на поддержку

### 3. Cloud-managed сервисы (serverless подход)
- **Описание**: AWS Lambda/Azure Functions, managed базы данных, SaaS мониторинг
- **Преимущества**: Минимальные операции, focus на бизнес-логике
- **Недостатки**: Меньший контроль, не демонстрирует навыки оркестрации и инфраструктуры

## Решение
Выбран **современный enterprise-стек** по следующим причинам:

### Kubernetes выбран потому что:
- **Демонстрация cloud-native экспертизы** – умение работать с production-grade оркестратором
- **Релевантность для российского рынка** – Yandex Cloud, SberCloud, Mail.ru Cloud Solutions используют Kubernetes
- **Доказательство DevOps/SRE навыков** – не только разработка, но и деплой, scaling, обслуживание
- **Подготовка к enterprise-ролям** – архитекторы в крупных компаниях должны понимать K8s

### Микросервисная архитектура (8 сервисов) потому что:
- **Демонстрация системной декомпозиции** – умение разбивать сложную систему на bounded contexts
- **Показ domain-driven design** – каждый сервис решает конкретную бизнес-задачу
- **Доказательство scalability знаний** – независимое масштабирование, fault isolation
- **Enterprise alignment** – российские компании активно мигрируют с монолитов на микросервисы

### Полный мониторинг (Prometheus/Grafana/Alertmanager) потому что:
- **Демонстрация observability mindset** – понимание, что monitoring не optional в production
- **Показ SRE принципов** – SLIs, SLOs, error budgets
- **Доказательство production experience** – знание, как отслеживать и реагировать на инциденты
- **Требование финансового сектора** – банки и крупные предприятия требуют comprehensive monitoring

### Mixed scripting (PowerShell/Bash/Python) потому что:
- **Демонстрация cross-platform экспертизы** – Windows (PowerShell) и Linux (Bash) среды
- **Прагматичный подход** – правильный инструмент для правильной задачи
- **Автоматизация CI/CD** – скрипты для deployment, тестирования, maintenance
- **Windows ecosystem relevance** – многие российские предприятия используют Windows серверы

## Последствия

### Положительные:
- **Портфолио как доказательство экспертизы** – показывает, что архитектор может спроектировать и реализовать сложную систему
- **Релевантность для enterprise-работодателей** – особенно для российских компаний (Yandex, банки, интеграторы)
- **Демонстрация полного цикла разработки** – от кода до мониторинга в production
- **Ответ на вопрос "оверэнжиниринг"** – сложность оправдана целями портфолио

### Отрицательные:
- **Высокая сложность поддержки** – требует больше времени и усилий
- **Потенциальное непонимание ревьюеров** – могут не оценить контекст портфолио
- **Риск "сложности ради сложности"** – если не документировать решения (решается через ADR)

### Нейтральные:
- **Обучение как побочный эффект** – углубленное понимание технологий через их практическое применение
- **База для будущих проектов** – готовые patterns и конфигурации можно reuse
