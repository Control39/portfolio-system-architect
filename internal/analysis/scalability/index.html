    Анализ масштабируемости

    Направление: Масштабируемость (Scalability) Дата: 2026-03-21 Аналитик: SourceCraft Code Assistant Agent

    1. Введение

    Цель анализа — оценить способность системы расти по нагрузке (горизонтально и вертикально), выявить узкие места и предложить архитектурные улучшения.

    Методология: - Обзор архитектуры на предмет stateless/stateful компонентов. - Анализ использования баз данных, кэшей, очередей. - Проверка поддержки горизонтального масштабирования (Kubernetes HPA, репликация). - Оценка производительности под нагрузкой (стресс-тесты, метрики).

    2. Архитектурные принципы

    2.1. Stateless vs Stateful

    Компонент Тип Масштабируемость
    auth-service Stateless ✅ Горизонтально
    decision-engine Stateless ✅ Горизонтально
    portfolio-organizer Stateless ✅ Горизонтально
    ml-model-registry Stateful (модели в памяти) ⚠️ Ограниченно
    system-proof Stateful (файлы на диске) ⚠️ Ограниченно
    PostgreSQL Stateful ✅ Вертикально + репликация

    Проблемы: - ml-model-registry хранит модели в памяти — репликация сложна. - system-proof зависит от локальной файловой системы.

    2.2. Микросервисы vs Монолит

    • Архитектура микросервисная, что способствует горизонтальному масштабированию.
    • Проблемы: Слишком мелкое дробление (некоторые сервисы выполняют тривиальные задачи).

    3. Базы данных и хранилища

    3.1. PostgreSQL

    • Используется как основное хранилище.
    • Проблемы: Нет репликации чтения, нет шардинга.
    • Рекомендация: Настроить read‑replicas, рассмотреть Citus для шардинга.

    3.2. Redis / Memcached

    • Не используется.
    • Рекомендация: Внедрить Redis для кэширования сессий, результатов запросов.

    3.3. Очереди сообщений

    • RabbitMQ (в arch-compass-framework) используется для асинхронных задач.
    • Проблемы: Нет кластеризации, single point of failure.

    4. Горизонтальное масштабирование

    4.1. Kubernetes HPA

    • Настроен для ml-model-registry (см. deployment/k8s/base/services/ml-model-registry/hpa.yaml).
    • Проблемы: Для остальных сервисов HPA отсутствует.
    • Рекомендация: Добавить HPA для всех stateless сервисов.

    4.2. Репликация сервисов

    • В deployment указаны replicas: 2 для некоторых сервисов.
    • Проблемы: Не все сервисы имеют несколько реплик.

    4.3. Балансировка нагрузки

    • Используется Kubernetes Service (ClusterIP) + Ingress (Nginx).
    • Проблемы: Нет настройки sticky sessions для stateful сервисов.

    5. Производительность под нагрузкой

    5.1. Стресс-тесты

    • В проекте нет автоматизированных нагрузочных тестов.
    • Рекомендация: Внедрить k6 или Locust, добавить в CI.

    5.2. Метрики масштабируемости

    Метрика Текущее значение Целевое
    RPS (запросов в секунду) Не измерено > 1000
    Latency p95 Не измерено < 200 мс
    Максимальное количество подключений БД Не измерено > 500

    6. Узкие места

    6.1. База данных

    • Единственный экземпляр PostgreSQL — bottleneck.
    • Отсутствие connection pooling.

    6.2. Файловые операции

    • system-proof работает с локальными файлами — не масштабируется.
    • Рекомендация: Перейти на объектное хранилище (S3‑совместимое).

    6.3. Внешние зависимости

    • Зависимость от внешних API (GigaChain, сторонние сервисы) может стать узким местом.
    • Рекомендация: Внедрить circuit breaker, кэширование, fallback.

    7. Выявленные проблемы

    7.1. Критические

    1. Отсутствие репликации БД — риск потери данных и downtime.
    2. Stateful сервисы не готовы к горизонтальному масштабированию (ml-model-registry, system-proof).
    3. Нет нагрузочного тестирования — неизвестны пределы системы.

    7.2. Высокие

    1. Отсутствие кэширования — излишняя нагрузка на БД.
    2. Нет HPA для ключевых сервисов (auth-service, decision-engine, portfolio-organizer).
    3. RabbitMQ без кластеризации — single point of failure.

    7.3. Средние

    1. Не настроены лимиты ресурсов (resources/limits) в Kubernetes — риск исчерпания ресурсов ноды.
    2. Отсутствие мониторинга метрик масштабируемости (RPS, latency, ошибки).
    3. Нет автоматического scaling на основе кастомных метрик (например, длина очереди).

    7.4. Низкие

    1. Не используется connection pooling для БД.
    2. Нет sticky sessions для stateful сервисов.
    3. Отсутствие документации по scaling procedures.

    8. Рекомендации

    8.1. Немедленные действия (1-2 дня)

    1. Настроить репликацию PostgreSQL (master‑slave).
    2. Добавить HPA для всех stateless сервисов.
    3. Внедрить Redis для кэширования сессий и тяжёлых запросов.

    8.2. Краткосрочные (1-2 недели)

    1. Перевести stateful сервисы на масштабируемые хранилища:
    2. ml-model-registry → использовать shared storage (NFS) или внешний registry (MLflow).
    3. system-proof → мигрировать на S3‑совместимое объектное хранилище.
    4. Кластеризовать RabbitMQ (зеркалирование очередей).
    5. Настроить мониторинг метрик масштабируемости (RPS, latency, ошибки) в Grafana.
    6. Провести нагрузочное тестирование с помощью k6, определить текущие пределы.

    8.3. Долгосрочные (1-2 месяца)

    1. Внедрить шардинг БД (Citus, Vitess) для горизонтального роста данных.
    2. Реализовать auto‑scaling на основе кастомных метрик (длина очереди, время обработки).
    3. Создать disaster recovery plan с multi‑region развёртыванием.
    4. Архитектурный рефакторинг — объединить слишком мелкие сервисы, выделить отдельные сервисы для тяжёлых операций.

    10. Заключение

    Система обладает потенциалом для горизонтального масштабирования благодаря микросервисной архитектуре, но критически страдает от stateful компонентов и отсутствия репликации БД. Реализация рекомендаций позволит уверенно расти под увеличивающейся нагрузкой и обеспечит высокую доступность.
