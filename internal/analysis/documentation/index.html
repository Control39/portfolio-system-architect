    Анализ документации

    Направление: Документация (Documentation) Дата: 2026-03-21 Аналитик: SourceCraft Code Assistant Agent

    1. Введение

    Цель анализа — оценить качество, полноту и актуальность документации проекта, выявить пробелы и предложить улучшения.

    Методология: - Обзор структуры директории docs/. - Проверка README файлов в корне и подпроектах. - Оценка наличия руководств для разработчиков, пользователей, API. - Анализ актуальности (соответствие коду).

    2. Общая структура документации

    2.1. Корневая документация

    Файл Назначение Состояние
    README.md Основное описание проекта ✅ Хорошо
    CONTRIBUTING.md Руководство для контрибьюторов ✅ Хорошо
    CODE_OF_CONDUCT.md Кодекс поведения ✅ Хорошо
    LICENSE Лицензия MIT ✅ Хорошо
    TODO.md Список задач ✅ Актуально
    QUICKSTART.md Быстрый старт ✅ Хорошо
    ONBOARDING.md Онбординг новых участников ✅ Хорошо

    docs/

    Проблемы: - Некоторые файлы дублируются (например, ARCHITECTURE.md в корне и в docs/). - Отсутствует единый навигационный индекс. - Документация API сгенерирована, но не обновляется автоматически.

    3. Документация для разработчиков

    3.1. Установка и настройка

    • QUICKSTART.md покрывает базовые шаги.
    • scripts/dev/setup-dev.ps1 — скрипт для Windows.
    • Проблемы: Нет инструкций для macOS/Linux.

    3.2. Архитектура и дизайн

    • ARCHITECTURE.md детально описывает компоненты.
    • diagrams/ содержит Mermaid-диаграммы.
    • Проблемы: Диаграммы не всегда синхронизированы с кодом.

    3.3. API документация

    • Используется Sphinx ( docs/api/ ).
    • Проблемы: Не все эндпоинты задокументированы, отсутствуют примеры запросов.

    3.4. Тестирование

    • Нет отдельного документа о тестировании, кроме CONTRIBUTING.md.
    • Рекомендация: Создать TESTING.md с описанием unit/integration/e2e тестов.

    4. Документация для пользователей

    4.1. Руководство пользователя

    • Отсутствует.
    • Рекомендация: Создать docs/user-guide/ с пошаговыми инструкциями.

    4.2. FAQ

    • docs/grants/FAQ.md существует, но только для грантов.
    • Рекомендация: Создать общий FAQ.

    4.3. Changelog

    • scripts/dev/CHANGELOG.md есть, но не интегрирован с релизами.
    • Рекомендация: Внедрить автоматическую генерацию changelog из тегов.

    5. Актуальность

    5.1. Сравнение с кодом

    Выборочная проверка:

    Документ Соответствие коду Комментарий
    ARCHITECTURE.md Частично Некоторые сервисы (например, auth-service) описаны, но их реализация изменилась.
    deployment/k8s-README.md Да Актуально.
    apps/portfolio-organizer/README.md Да Актуально.
    apps/decision-engine/README.md Нет Устарело (описывает старую версию).

    5.2. Устаревшие документы

    • docs/archive/ — помечены как архив, но могут вводить в заблуждение.
    • docs/grants/grant-ready.md — устаревшие даты.

    6. Форматы и инструменты

    6.1. Используемые форматы

    • Markdown (основной).
    • Mermaid для диаграмм.
    • Sphinx для API.

    6.2. Инструменты генерации

    • scripts/generate-site.sh — генерация статического сайта.
    • tools/generate_website.py — скрипт для создания веб-сайта.
    • Проблемы: Нет автоматической публикации документации (например, через GitHub Pages).

    7. Выявленные проблемы

    7.1. Критические

    1. Отсутствие руководства пользователя — новые пользователи не знают, как использовать систему.
    2. Дублирование архитектурной документации — приводит к рассинхронизации.
    3. Устаревшая документация по некоторым сервисам (decision-engine → decision-engine, arch-compass-framework → infra-orchestrator - переименованы, требуется обновление ссылок).

    7.2. Высокие

    1. Нет единого навигационного индекса — сложно найти нужный документ.
    2. Документация API неполная — отсутствуют примеры, описание ошибок.
    3. Отсутствие автоматической публикации — документация не доступна онлайн.

    7.3. Средние

    1. Нет инструкций для macOS/Linux.
    2. Не все диаграммы актуальны.
    3. FAQ только для грантов.

    7.4. Низкие

    1. Changelog не интегрирован с релизами.
    2. Нет документации по тестированию.
    3. Отсутствие глоссария терминов.

    8. Рекомендации

    8.1. Немедленные действия (1-2 дня)

    1. Создать навигационный индекс (docs/INDEX.md) с картой документации.
    2. Обновить устаревшие README (decision-engine, arch-compass-framework).
    3. Удалить дубликаты (оставить один ARCHITECTURE.md в docs/).

    8.2. Краткосрочные (1-2 недели)

    1. Написать руководство пользователя (user-guide) с пошаговыми сценариями.
    2. Дополнить документацию API примерами запросов/ответов, кодами ошибок.
    3. Создать общий FAQ на основе частых вопросов.
    4. Настроить автоматическую публикацию через GitHub Pages / MkDocs.
    5. Добавить инструкции для macOS/Linux в QUICKSTART.md.

    8.3. Долгосрочные (1-2 месяца)

    1. Внедрить автоматическую синхронизацию диаграмм с кодом (например, через PlantUML).
    2. Интегрировать changelog с релизами (использовать git-cliff или standard-version).
    3. Создать глоссарий терминов и справочник архитектурных решений (ADR).
    4. Регулярные аудиты документации (раз в квартал).

    10. Заключение

    Документация проекта обширна, но страдает от дублирования, устаревания и отсутствия целостной навигации. Критически важно создать руководство пользователя и наладить процессы поддержания актуальности. Реализация рекомендаций сделает документацию полезным активом, а не обузой.
