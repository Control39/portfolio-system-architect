# ADR-017: mcp_server Coverage Target Decision

**Дата:** 17 мая 2026 г.
**Статус:** Принято (Accepted)
**Автор:** Control39
**Решение:** Оставить покрытие mcp_server на уровне 54% (v1.0.0-rc1)

---

## 📋 Контекст

### Проблема

Модуль `mcp_server` содержит инструменты для работы с файловой системой, Git, ChromaDB и мониторингом. Все инструменты обернуты в кастомные MCP-декораторы `@mcp_server.tool()`, что создаёт сложности для юнит-тестирования.

### Текущее состояние

- **Покрытие:** 54.12% (151 тест, 146 passed)
- **Целевой показатель:** 80%
- **Оставшиеся 26%** требуют тестирования MCP-обёрток

### Затраты на достижение 80%

| Покрытие | Тестов | Время | Тип тестов |
|----------|--------|-------|------------|
| 35% → 54% | 115 | 3 часа | Бизнес-логика ✅ |
| 54% → 80% | ~100 | 3-4 часа | Моки декораторов ⚠️ |

**Анализ:** Следующие 26% покрытия потребуют ~80% дополнительных усилий.

---

## 💡 Решение

### Принятое решение

**Оставить покрытие mcp_server на уровне 54% для релиза v1.0.0-rc1.**

### Обоснование

1. **Закон убывающей отдачи**
   - Первые 54% достигнуты за 3 часа (бизнес-логика)
   - Следующие 26% потребуют 3-4 часа (сложные моки)
   - ROI: 1% покрытия = 10 минут vs 1% = 90 минут

2. **Критичность для production**
   - Бизнес-логика покрыта: ✅
   - Основные сценарии протестированы: ✅
   - Интеграция с другими сервисами работает: ✅
   - Критические уязвимости отсутствуют: ✅

3. **Архитектурный риск**
   - Тесты для MCP-декораторов будут хрупкими
   - Ломаются при изменении FastMCP API
   - Требуют поддержки сложных моков

4. **Приоритеты проекта**
   - 11 из 15 сервисов имеют ≥80% ✅
   - Persistence слой (PostgreSQL) важнее
   - E2E тесты критичны для CI/CD

---

## 🔄 Альтернативы

### A. Дожать до 80% сейчас
**Плюсы:**
- Все сервисы ≥80%
- Чистая метрика для резюме

**Минусы:**
- 3-4 часа на хрупкие тесты
- Отвлечение от критичных задач
- Риск выгорания

**Вердикт:** ❌ Не рекомендуется

### B. Рефакторинг архитектуры (v1.1)
**План:**
1. Вынести бизнес-логику из-под декораторов
2. Создать чистые функции для тестирования
3. Обернуть в MCP-декораторы только на уровне API

**Оценка:** 6-8 часов
**Риск:** Средний (требует изменений production кода)

**Вердикт:** ✅ Рекомендуется для v1.1

### C. Принять 54% как достаточное
**План:**
- Зафиксировать технический долг
- Сфокусироваться на других компонентах
- Вернуться при наличии ресурсов

**Вердикт:** ✅ Выбрано для v1.0.0-rc1

## 📊 Последствия

### Положительные

✅ **Экономия времени:** 3-4 часа на другие задачи
✅ **Документированное решение:** ADR + TESTING_STRATEGY.md
✅ **Фокус на ценности:** бизнес-логика покрыта
✅ **Прозрачность:** любое лицо может понять обоснование

### Отрицательные

⚠️ **Технический долг:** 46% кода без тестов
⚠️ **Риск регрессии:** изменения в MCP-декораторах могут сломать инструменты
⚠️ **Метрика ниже цели:** 54% вместо 80%

### Нейтральные

➖ **Восприятие:** требует объяснения на собеседовании
➖ **CI/CD:** текущая конфигурация не требует 80% для всех модулей

---

## 🎯 Критерии завершения (для v1.1)

Решение пересматривается при выполнении **одного из условий:**

1. **Выполнен рефакторинг архитектуры** (Вариант B)
   - Бизнес-логика вынесена из-под декораторов
   - Покрытие ≥80% достижимо за ≤2 часов

2. **Изменились требования**
   - Инвесторы/HR требуют ≥80% для всех модулей
   - CI/CD блокирует релиз с <80%

3. **Появились ресурсы**
   - Выделено 4+ часа на доведение mcp_server
   - Приоритет сменился с "низкого" на "высокий"

## 🗣️ Как презентовать

### На собеседовании

> «В модуле mcp_server инструменты обернуты в кастомные MCP-декораторы, что усложняет юнит-тестирование. Я провела анализ затрат: достичь 80% покрытия потребовало бы 3-4 часов на написание хрупких моков.
>
> Приняла архитектурное решение выпустить релиз с 54% покрытием (146 тестов), зафиксировав обоснование в ADR. Это позволило сфокусироваться на критичных для production компонентах: persistence-слое и E2E-тестах.
>
> Покрытие тестами — инструмент, а не цель. Важнее демонстрировать системное мышление и умение принимать взвешенные решения на основе данных.»

### В резюме

> **mcp_server (54% покрытие)**
> - 146 тестов покрывают бизнес-логику и основные сценарии
> - Архитектурное решение: отложено до v1.1 из-за сложности моков MCP-декораторов
> - Задокументировано в ADR-017 и TESTING_STRATEGY.md
