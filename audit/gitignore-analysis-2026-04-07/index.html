    Анализ необходимости исключения конфигураций AI-агентов из Git

    Дата: 2026-04-07 Автор: SourceCraft Code Assistant Agent Цель: Определить, нужно ли исключать конфигурации AI-агентов из Git, и дать рекомендации

    Текущая ситуация

    В .gitignore исключены следующие директории AI-агентов:

    codeassistant/
.continue/
.gigaide/
.sourcecraft/
.ai-config/
.ai-config-gui/

    Что содержится в этих директориях (предположительно)

    1. codeassistant/ - Code Assistant конфигурации

    • Навыки (skills/) - переиспользуемые инструкции для агентов
    • Контекст (context.md) - контекст проекта для агентов
    • Кастомные режимы (custom_modes.yaml) - специализированные режимы работы
    • Правила (rules/) - правила игнорирования и доступа
    • MCP конфигурации (mcp.json) - настройки MCP серверов

    2. .continue/ - Continue.dev конфигурации

    • Конфигурации агентов (agents/) - настройки для разных сценариев
    • Модели AI - настройки подключения к моделям (OpenAI, Anthropic и др.)
    • MCP серверы - конфигурации внешних серверов

    3. .gigaide/ - GigaIDE конфигурации

    • Настройки IDE расширений
    • Конфигурации рабочих пространств

    4. .sourcecraft/ - SourceCraft конфигурации

    • Настройки интеграции с SourceCraft платформой
    • Конфигурации грантов и проектов

    5. .ai-config/ и .ai-config-gui/ - AI Config Manager

    • Генераторы конфигураций для разных инструментов
    • Шаблоны и настройки

    Аргументы ЗА исключение из Git

    1. Приватность и безопасность

    • API ключи: Конфигурации могут содержать API ключи к сервисам AI
    • Персональные настройки: Индивидуальные предпочтения пользователя
    • Локальные пути: Абсолютные пути, специфичные для конкретной машины

    2. Избыточность и шум

    • Авто-генерация: Многие конфигурации генерируются автоматически
    • Частые изменения: Конфигурации часто меняются в процессе работы
    • Версионность не нужна: Не требуется отслеживать историю изменений конфигураций

    3. Конфликты при совместной работе

    • Разные среды: У разных разработчиков разные настройки
    • Конфликты слияния: Частые конфликты при работе в команде
    • Локальные эксперименты: Экспериментальные настройки не должны влиять на других

    Аргументы ПРОТИВ исключения из Git

    1. Воспроизводимость среды

    • Настройки проекта: Конфигурации - часть проекта, нужны для воспроизведения
    • Совместимость: Гарантия, что у всех одинаковые настройки
    • Onboarding: Новые участники получают готовую конфигурацию

    2. Обнаруженные проблемы

    • Устаревшие настройки: GPT-5 и другие несуществующие модели
    • Конфликты версий: Разные версии конфигураций у разных инструментов
    • Отсутствие аудита: Невозможно проверить на безопасность и актуальность

    3. Career Autopilot требования

    • Анализ компетенций: Для автоматического обнаружения маркеров нужен доступ к конфигурациям
    • Генерация портфолио: Конфигурации показывают используемые инструменты и подходы
    • Демонстрация экспертизы: Умение настраивать AI-агенты - часть экспертизы

    Анализ по типам файлов

    Следует исключать (оставить в .gitignore):

    1. Файлы с секретами:
    2. API ключи (apiKey, token, secret)
    3. Личные access tokens

    4. Креденшиалы облачных сервисов

    5. Локальные настройки:

    6. Абсолютные пути к локальным директориям
    7. Настройки конкретной IDE/редактора

    8. Персональные предпочтения пользователя

    9. Авто-генерируемые файлы:

    10. Кэши
    11. Логи
    12. Временные файлы

    Следует включить в Git (убрать из .gitignore):

    1. Шаблоны конфигураций:
    2. Примеры конфигураций (.example.yaml, .template.yaml)
    3. Документация по настройке

    4. Best practices

    5. Конфигурации проекта:

    6. Структура навыков (skills/)
    7. Контекст проекта (context.md)

    8. ADR (Architectural Decision Records) для AI-агентов

    9. Интеграционные конфигурации:

    10. Настройки MCP серверов (без секретов)
    11. Конфигурации CI/CD для AI-агентов
    12. Скрипты развертывания

    Рекомендации для проекта "portfolio-system-architect"

    Учитывая цели проекта:

    1. Грант SourceCraft Open Source: Нужно демонстрировать экспертизу в настройке AI-агентов
    2. Career Autopilot: Нужен доступ к конфигурациям для анализа компетенций
    3. Воспроизводимость: Другие должны иметь возможность воспроизвести вашу среду

    Предлагаемое решение:

    1. Создать два уровня конфигураций:
    2. config/ai/ - публичные конфигурации (в Git)

    3. config/ai/local/ - локальные конфигурации (в .gitignore)

    4. Использовать шаблоны с подстановкой: ```yaml # config/ai/continue.template.yaml models:

      • name: "gpt-4" provider: "openai" apiKey: "${OPENAI_API_KEY}" # Заменяется через переменные окружения ```

    5. Документировать процесс настройки:

    6. docs/ai-agents-setup.md - инструкция по настройке
    7. scripts/setup-ai-configs.py - скрипт инициализации

    Конкретные рекомендации для .gitignore

    Текущие правила (оставить):

    # Секреты
*.key
*.pem
*.p12
*.pfx
*credentials*.json

# Локальные настройки
agents/local/config/*.local.yaml
agents/gigacode/config/*.local.yaml

# Авто-генерация
*.log
*.csv
*-report.json

    Новые правила (добавить/изменить):

    # Вместо полного исключения директорий
# codeassistant/  # ← закомментировать

# Разрешить конкретные файлы
!codeassistant/skills/
!codeassistant/context.md
!codeassistant/custom_modes.yaml

# Исключить только секреты
codeassistant/*.local.yaml
codeassistant/*-secret.yaml

# Аналогично для других директорий
!.continue/agents/*.template.yaml
!.continue/config.example.yaml
.continue/*.local.yaml

    План действий

    Фаза 1: Аудит текущих конфигураций

    1. Временно разрешить доступ к конфигурациям
    2. Проанализировать содержимое на:
    3. Секреты (API ключи, токены)
    4. Локальные пути
    5. Устаревшие настройки (GPT-5)
    6. Создать отчет

    Фаза 2: Реорганизация

    1. Создать структуру config/ai/ с публичными конфигурациями
    2. Перенести безопасные конфигурации в публичную директорию
    3. Создать шаблоны для секретных настроек

    Фаза 3: Обновление .gitignore

    1. Закомментировать полные исключения директорий
    2. Добавить выборочные правила
    3. Протестировать, что секреты не попадают в Git

    Фаза 4: Документация

    1. Создать инструкцию по настройке
    2. Добавить скрипты инициализации
    3. Обновить README проекта

    Риски и ограничения

    Риски:

    1. Утечка секретов: Если неправильно настроить .gitignore
    2. Конфликты: При переходе на новую структуру
    3. Время: Реорганизация займет время

    Ограничения:

    1. Текущая блокировка: Невозможно проанализировать конфигурации без доступа
    2. Неизвестное содержимое: Не знаем, что именно в конфигурациях
    3. Зависимости: Другие инструменты могут полагаться на текущую структуру

    Выводы и рекомендации

    Для проекта "portfolio-system-architect":

    1. Рекомендуется убрать полные исключения директорий AI-агентов из .gitignore
    2. Перейти на выборочное исключение только секретных и локальных файлов
    3. Создать публичные конфигурации как часть демонстрации экспертизы
    4. Использовать шаблоны и переменные окружения для секретов

    Преимущества этого подхода:

    1. Демонстрация экспертизы: Конфигурации AI-агентов показывают ваш уровень
    2. Воспроизводимость: Другие могут воспроизвести вашу среду
    3. Аудит и безопасность: Возможность проверки на устаревшие и небезопасные настройки
    4. Career Autopilot: Доступ к конфигурациям для анализа компетенций

    Следующие шаги:

    1. Получить доступ к текущим конфигурациям для аудита
    2. Создать план миграции на новую структуру
    3. Поэтапно реализовать изменения, начиная с наименее критичных конфигураций

    Рекомендация: Начать с временного разрешения доступа к .continue/agents/new-config.yaml для исправления критической проблемы с GPT-5, затем постепенно реорганизовывать остальные конфигурации.
