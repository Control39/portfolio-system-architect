    ADR-016: Стандартизация документации микросервисов через шаблон

    Дата принятия: 15 мая 2026 г. Статус: Принято ✅ Автор: [YourName] Решение: Шаблон docs/README_TEMPLATE.md + валидатор scripts/check_ports.py

    📋 Контекст

    Проект содержит 14 микросервисов в директории apps/. Каждый сервис имеет собственный README.md с описанием функциональности, API и конфигурации.

    Проблема

    1. Расхождение документации от кода:
    2. apps/README.md показывает порты, которые не соответствуют docker-compose.yml
    3. Пример: ml-model-registry в README = порт 8001, в docker-compose = порт 8001 (конфликт с decision-engine)

    4. Интервьюеры проверяют детали → теряется доверие к репозиторию

    5. Отсутствие единого стандарта:

    6. Каждый README написан по-разному (разная структура, разные разделы)
    7. Нет обязательных разделов (API, безопасность, тестирование)

    8. Сложно онбордить новых контрибьюторов

    9. Ручная проверка:

    10. Нет автоматической валидации актуальности
    11. Расхождения обнаруживаются только при ручном аудите или в процессе интервью

    Влияние

    • ❌ Потеря доверия: Технические интервьюеры видят несоответствия → сомнения в качестве
    • ❌ Сложность поддержки: Новые контрибьюторы тратят время на поиск актуальной информации
    • ❌ Риск деплоя: Неправильные порты/маршруты → нерабочая инфраструктура

    🎯 Решение

    1. Создать унифицированный шаблон docs/README_TEMPLATE.md

    Структура шаблона:

    | Раздел | Содержание | |------------| | 🎯 Назначение | 1-2 предложения о бизнес-логике | | 🏗️ Архитектура | Технологии, зависимости, структура каталогов | | 🚀 Quick Start | Запуск через Docker Compose и локально | | 🔌 API Контракты | Таблица эндпоинтов + примеры запросов | | 🛡️ Безопасность | Чек-лист мер защиты (SSRF, маскирование, JWT) | | 🧪 Тестирование | Команды запуска тестов, таблица покрытия | | 📊 Мониторинг | Prometheus/Grafana, логи, health checks | | 🔧 Конфигурация | Переменные окружения, пример .env | | 📝 История изменений | Таблица версий | | 🤝 Вклад | Задачи для контрибьюторов |

    Применено к: - ✅ apps/decision_engine/README.md - ✅ apps/it_compass/README.md - ✅ apps/career_development/README.md

    Осталось применить к: - ⏳ apps/ml_model_registry/README.md - ⏳ apps/portfolio_organizer/README.md - ⏳ apps/system_proof/README.md - ⏳ apps/job-automation-agent/README.md - ⏳ apps/knowledge_graph/README.md - ⏳ apps/thought-architecture/README.md - ⏳ apps/infra-orchestrator/README.md - ⏳ apps/mcp_server/README.md - ⏳ apps/ai-config-manager/README.md - ⏳ apps/cognitive-agent/README.md

    2. Создать валидатор scripts/check_ports.py

    Скрипт автоматизирует проверку согласованности портов:

    # Парсит docker-compose.yml → извлекает порты из Traefik labels
# Парсит apps/README.md → извлекает порты из таблицы
# Сравнивает → сообщает расхождения и пропуски
# Режим --strict для CI/CD (возвращает exit code 1 при ошибках)

    Пример вывода:

    ============================================================
🔍 Проверка согласованности портов
============================================================
✅ Согласовано (7 сервисов):
   • auth_service: port 8100
   • decision_engine: port 8001
   • ml_model_registry: port 8001
   ...
============================================================
✅ Проверка пройдена
============================================================

    3. Интегрировать в workflow

    Инструмент Конфигурация
    Pre-commit .pre-commit-config.yaml → hook check-ports
    Makefile make check-ports
    CI/CD .github/workflows/ci.yml → python scripts/check_ports.py --strict
    Документация CONTRIBUTING.md → чек-лист перед PR

    🔄 Альтернативы

    | Вариант | Плюсы | Минусы | Почему не выбран | |---------|----------------| | Оставить как есть | Быстро, без изменений | Документация устаревает, теряется доверие | ❌ Не масштабируется | | Ручная проверка | Простота, нет скриптов | Требует времени, человеческие ошибки | ❌ Ненадёжно | | Внешние инструменты (Swagger, Redoc) | Автоматическая генерация | Не покрывает бизнес-логику, методологию | ❌ Частично решает | | Шаблон + скрипт | Автоматизация, консистентность, измеримость | Требует начальных усилий, поддержка | ✅ Выбрано |

    ⚖️ Последствия

    Положительные

    • ✅ Документация всегда актуальна — скрипт предотвращает расхождения
    • ✅ Интервьюеры видят точную информацию — доверие к репозиторию
    • ✅ Быстрый онбординг — новые контрибьюторы знают, что искать
    • ✅ CI/CD предотвращает регрессию — pre-commit hook ловит ошибки до коммита

    Негативные

    • ⚠️ Нужно обновить 10+ README — ручная работа (~2-3 часа)
    • ⚠️ Поддержка скрипта — при изменении структуры docker-compose.yml нужно обновлять парсер
    • ⚠️ Шаблон может стать «слишком строгим» — некоторые сервисы могут не подходить под структуру

    Риски

    • ⚠️ Скрипт может не учесть новые форматы — например, если добавить новый сервис без порта
    • ⚠️ Ложные срабатывания — если docker-compose.yml и README.md намеренно отличаются (редко)
    • ⚠️ Конфликт портов через Traefik — скрипт проверяет только порты, но не маршруты

    📊 Примеры

    До (проблема)

    ## Приложения
| Сервис | Порт |
|--------|------|
| ml-model-registry | 8001 |

    Проблема: decision-engine тоже использует 8001 → конфликт в Traefik.

    После (решение)

    ## Приложения
| Сервис | Порт | Маршрут (Traefik) |
|--------|------|-------------------|
| ml_model_registry | 8001 | `/ml-registry` |
| decision_engine | 8001 | `/decision-engine` |

    Решение: Один порт, разные маршруты через Traefik.

    Валидация

    python scripts/check_ports.py --strict
# ✅ Согласовано (7 сервисов)

    ✅ Принятое решение

    Использовать унифицированный шаблон docs/README_TEMPLATE.md для всех микросервисов + автоматическую валидацию через scripts/check_ports.py.

    Ключевой принцип:

    Документация должна быть синхронизирована с кодом. Автоматическая валидация предотвращает расхождение данных.
