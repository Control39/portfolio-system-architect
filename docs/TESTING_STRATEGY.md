## Обоснование решения

### Почему не доведено до 80%?

**Техническая причина:**
Инструменты mcp_server обернуты в кастомные MCP-декораторы `@mcp_server.tool()`, что требует:
1. Инициализации полного MCP-сервера для тестирования
2. Сложных моков для FastMCP instance
3. Интеграционных тестов вместо юнит-тестов

**Анализ отдачи:**
| Покрытие | Добавлено тестов | Затрачено времени | Ценность |
|----------|------------------|-------------------|----------|
| 35% → 51% | 86 | 2 часа | Высокая (бизнес-логика) |
| 51% → 54% | 15 | 1 час | Средняя (концепции) |
| 54% → 80% | ~100 | 3-4 часа | Низкая (моки декораторов) |

**Вывод:** Следующие 26% покрытия потребуют ~80% дополнительных усилий.

## План завершения (v1.1)

### Вариант A: Рефакторинг архитектуры (рекомендуется)
**Цель:** Упростить тестирование инструментов

**Задачи:**
1. Вынести бизнес-логику из-под декораторов
2. Создать чистые функции для тестирования
3. Обернуть в MCP-декораторы только на уровне API

**Оценка:** 6-8 часов
**Риск:** Средний (требует изменений в production коде)

### Вариант B: Интеграционные тесты
**Цель:** Написать полные моки для MCP-сервера

**Задачи:**
1. Создать `MockFastMCP` class
2. Тестировать инструменты через mock-сервер
3. Добавить 100+ интеграционных тестов

**Оценка:** 4-6 часов
**Риск:** Высокий (хрупкие тесты, ломаются при изменении MCP API)

### Вариант C: Принять 54% как достаточное
**Цель:** Сфокусироваться на других компонентах

**Обоснование:**
- 11 из 12 сервисов имеют ≥80%
- mcp_server работает стабильно (146 тестов)
- Критичная бизнес-логика покрыта

**Решение:** Зафиксировать как технический долг, вернуться при необходимости
