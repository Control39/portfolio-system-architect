## 🎯 Стратегия разделения ответственности

### 1. architect-analize (Оставить без изменений)

**Зона ответственности:**
- Системное мышление и IT-Compass методология
- ADR и архитектурные trade-offs
- Feedback loops и самообучение системы
- Когнитивные паттерны

**Чёткое отличие от `repo-quality-auditor`:**
| architect-analize | repo-quality-auditor |
|-------------------|----------------------|
| "Почему выбраны эти решения?" | "Соответствует ли структура стандартам?" |
| "Как система эволюционирует?" | "Все ли тесты проходят?" |
| "Где паттерны системного мышления?" | "Все ли бейджи актуальны?" |

**Решение:** ✅ Оставить как есть

---

### 2. integrity-checker (Уточнить границы)

**Текущее:**
- Проверка ссылок (lychee)
- Проверка бейджей
- Поиск дубликатов файлов
- Проверка импортов

**Дублирование с `repo-quality-auditor`:**
- Проверка ссылок ✅ (есть в обоих)
- Проверка бейджей ✅ (есть в обоих)

**Уникальное:**
- Автоматизация через скрипты (`scripts/analyze_links.py`)
- Поиск дубликатов по хешу

**Решение:**
```
integrity-checker → сосредоточиться на АВТОМАТИЗАЦИИ:
- Скрипты для автоматической проверки (lychee, py_compile)
- CI/CD интеграция
- Auto-fix сломанных ссылок
- Дубликаты файлов (уникально)

repo-quality-auditor → сосредоточиться на КАЧЕСТВЕ:
- Архитектурные стандарты
- Покрытие тестами
- Документация
- Бейджи
```

**Действия:**
1. Удалить из `integrity-checker` проверки ссылок и бейджей (перенести в `repo-quality-auditor`)
2. Добавить в `integrity-checker` уникальные проверки (дубликаты, авто-fix)
3. Обновить SKILL.md с чёткими границами

### 3. code + security → объединить или разделить

**Вариант A: Объединить в code-security**

```
codeassistant/skills/code-security/
├── SKILL.md          # Единые инструкции
├── architecture.md   # Архитектурный код-ревью (из code/)
├── security.md       # Security by Design (из security/)
└── tools/
    ├── linting.py    # Black, Ruff, MyPy
    ├── security.py   # Bandit, Trivy, secret detection
    └── tests.py      # pytest, coverage
```

**Вариант B: Разделить чётко**

```
code (архитектурный код-ревью):
- Соответствие ADR
- Интерфейсы и контракты
- Наблюдаемость (логи, метрики, health checks)
- Тестируемость

security (безопасность):
- SQL injection защита
- Секреты (Sealed Secrets, env)
- CI/CD безопасность
- Network policies
```

**Рекомендация:** Вариант B (разделить), так как зоны ответственности разные
