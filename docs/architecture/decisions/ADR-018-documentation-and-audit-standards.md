# ADR-018: Стандарты документации и аудита сервисов

## Status

Принято (Accepted)

## Дата

19 мая 2026 г.

## Автор

Control39

## Context

### Проблема

1. **Фрагментарность документации:** Корневой `README.md` перегружен информацией для разных
   аудиторий (HR, техлиды, рекрутеры), что ухудшает user experience.
1. **Несоответствие документации и кода:** В `README.md` сервисов часто описана устаревшая или
   неактуальная реализация.
1. **Отсутствие единого плана развития:** Нет централизованной дорожной карты для улучшения
   сервисов.
1. **Необходимость масштабируемого аудита:** Требуется системный подход к верификации 15+ сервисов.

### Цель

Закрепить архитектурные решения по структуре документации и процессу аудита, обеспечив единообразие,
актуальность и системность во всей экосистеме.

## Decision

### 1. Разделение README по аудитории

Создать отдельные документы в `docs/` для разных групп:

| Аудитория | Документ | Назначение | |----------|----------|------------| | **Рекрутеры и HR** |
`docs/FOR_HR.md` | Фокус на компетенциях, ролях, объективных маркерах | | **Технические лиды** |
`ARCHITECTURE.md` | Глубокая архитектура системы, стек технологий, ADR | | **Грантовые комитеты** |
`docs/GRANT_PROPOSAL.md` | Уникальность методологии, потенциал для сообщества | | **Контрибьюторы**
| `docs/QUICK_START.md` | Инструкции по настройке и внесению вклада |

Корневой `README.md` используется как **навигационный хаб** с кратким введением и ссылками.

### 2. Процесс верификации сервисов

Запустить проект `apps-audit` для пошаговой проверки каждого сервиса в `apps/`.

**Этапы аудита одного сервиса:**

1. **Сверка `README.md` с кодом:** Проверить, соответствует ли документация актуальной реализации.
1. **Анализ зависимостей:** Просмотреть все импорты и зависимости.
1. **Поиск расхождений:** Исправить несоответствия в `README.md`.
1. **Добавление `Roadmap`:** Добавить в `README.md` секцию с планом развития.
1. **Создание диаграммы:** Добавить `mermaid.js` диаграмму внутренней архитектуры сервиса.
1. **Стандартизация кода:** Проверить код на лишние элементы (эмодзи, избыточные комментарии).
1. **Создание отчета:** Зафиксировать результаты в `.reports/.audit/`.

### 3. Стандартизация кода и README

- **Эмодзи:** Допускаются только в заголовках секций для визуальной навигации. Удалить из кода и
  обычного текста.
- **Лицензия:** Убедиться, что каждый сервис имеет `LICENSE` файл.
- **Форматирование:** Соблюдать `black` для Python и `prettier` для Markdown.

## Альтернативы

### A. Оставить как есть (No Change)

**Плюсы:** Меньше усилий. **Минусы:** Плохой UX, устаревшая документация, сложность масштабирования.
**Вердикт:** ❌ Не рекомендуется

### B. Создать только дорожные карты

**Плюсы:** Улучшает прозрачность. **Минусы:** Не решает проблему фрагментарности и несоответствия.
**Вердикт:** ❌ Частичное решение

### C. Принятое решение (A + B)

**Плюсы:** Решает все ключевые проблемы, создает масштабируемый процесс. **Вердикт:** ✅ Выбрано

## Consequences

### Positive

✅ **Улучшенный UX:** Каждая аудитория находит свою информацию быстро. ✅ **Актуальность:**
Документация всегда соответствует коду. ✅ **Прозрачность:** Явная дорожная карта повышает доверие. ✅
**Масштабируемость:** Процесс аудита можно применить ко всем сервисам. ✅ **Повышение качества:**
Стандартизация кода и документации.

### Отрицательные

⚠️ **Временные затраты:** Требует времени на первоначальный аудит. ⚠️ **Риск неполноты:** Процесс
требует дисциплины для завершения всех сервисов.

## Решение о принятии

Решение принято на основании высокой ценности для долгосрочной поддержки проекта, улучшения
воспринимаемой ценности и создания системы для постоянного улучшения.
