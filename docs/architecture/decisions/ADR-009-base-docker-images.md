# ADR-009: Создание базовых Docker образов для стандартизации разработки

**Статус**: Принято
**Дата**: 2026-03-31
**Автор**: Архитектурная команда
**Номер**: ADR-009

## Контекст
В проекте portfolio-system-architect используется несколько микросервисов и компонентов, каждый со своими Dockerfile. Это приводит к следующим проблемам:

1. **Дублирование кода**: Каждый Dockerfile содержит повторяющиеся инструкции (установка зависимостей, настройка окружения)
2. **Несогласованность версий**: Разные сервисы используют разные версии базовых образов и зависимостей
3. **Длительное время сборки**: Каждый сервис выполняет одни и те же шаги при сборке
4. **Сложность обновлений**: Обновление базовых зависимостей требует изменений во всех Dockerfile
5. **Размер образов**: Отсутствие оптимизации приводит к большим размерам конечных образов

Текущая архитектура включает Python-сервисы, PowerShell скрипты и различные инструменты, требующие стандартизированного подхода к контейнеризации.

## Рассмотренные варианты
1. **Базовые образы для каждого языка/стека**: Создание собственных базовых образов для Python, Node.js, PowerShell
2. **Многоступенчатые сборки в каждом сервисе**: Оптимизация каждого Dockerfile индивидуально
3. **Использование официальных образов с дополнениями**: Наследование от официальных образов с добавлением общих зависимостей
4. **Docker layer caching с общими шагами**: Вынесение общих шагов в отдельные слои через Docker buildkit
5. **Отказ от оптимизации**: Сохранение текущего подхода с дублированием

## Решение
Выбрали создание базовых образов для каждого основного технологического стека:

1. **Python базовый образ** (`docker/base-images/python/Dockerfile`):
   - Наследование от `python:3.11-slim`
   - Установка системных зависимостей (build-essential, curl, git)
   - Предустановка common Python пакетов (pytest, black, flake8, mypy)
   - Настройка рабочего каталога и пользователя
   - Многоступенчатая сборка для оптимизации размера

2. **Стандартизированная структура**:
   - Все базовые образы размещаются в `docker/base-images/`
   - Каждый образ имеет README с документацией
   - Версионирование через теги (например, `portfolio-base-python:1.0`)
   - Автоматическая сборка через CI/CD

3. **Принципы разработки**:
   - Минимальный размер образов
   - Безопасность (non-root пользователи)
   - Воспроизводимость (фиксированные версии зависимостей)
   - Поддержка development и production сценариев

4. **Интеграция с сервисами**:
   - Сервисы наследуются от базовых образов
   - Добавляют только специфичные зависимости
   - Используют общие best practices

## Последствия
### Положительные:
- **Сокращение дублирования**: Единый источник истины для общих зависимостей
- **Ускорение сборки**: Кэширование базовых слоёв ускоряет сборку сервисов
- **Стандартизация**: Единые версии зависимостей и конфигурации безопасности
- **Упрощение обновлений**: Обновление зависимостей в одном месте
- **Уменьшение размера образов**: Оптимизация через многоступенчатые сборки

### Отрицательные:
- **Дополнительная сложность**: Управление версиями базовых образов
- **Начальные затраты**: Создание и поддержка базовых образов
- **Обучение команды**: Новые разработчики должны знать о базовых образах
- **Зависимость от внутренней инфраструктуры**: Необходимость сборки и хранения образов

### Нейтральные:
- **Поэтапный переход**: Существующие сервисы могут мигрировать постепенно
- **Обратная совместимость**: Базовые образы поддерживают существующие сервисы
- **Гибкость**: Возможность создавать специализированные варианты базовых образов
