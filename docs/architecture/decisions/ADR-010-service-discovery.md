﻿# ADR-008: Внедрение Service Discovery с использованием Consul

**Статус**: Принято **Дата**: 2026-03-31 **Автор**: Архитектурная команда **Номер**: ADR-008

## Context

В текущей архитектуре проекта portfolio-system-architect используется статическая конфигурация
сервисов через Docker Compose с хардкодом адресов и портов. Это создаёт несколько проблем:

1. **Хрупкость конфигураций**: При изменении портов или имён сервисов необходимо обновлять все
   зависимые конфигурации
1. **Отсутствие динамического обнаружения**: Новые экземпляры сервисов не могут быть автоматически
   обнаружены
1. **Сложность масштабирования**: Ручное управление адресами при горизонтальном масштабировании
1. **Отсутствие health checks**: Нет автоматического обнаружения неработоспособных сервисов

Текущая архитектура включает микросервисы (career-development, thought-architecture, etc.), API
Gateway (Traefik) и требует надёжного механизма обнаружения сервисов.

## Considered Options

1. **Consul** - полнофункциональная система service discovery с поддержкой health checks,
   KV-хранилищем и DNS-интерфейсом
1. **etcd** - распределённое key-value хранилище, используемое в Kubernetes, но требует
   дополнительных компонентов для service discovery
1. **ZooKeeper** - распределённый координатор сервисов, более сложный в настройке и обслуживании
1. **DNS-based discovery** - использование DNS SRV записей, ограниченная функциональность
1. **Client-side discovery** - встроенная логика обнаружения в клиентах, увеличивает сложность
   клиентов

## Decision

Выбрали Consul по следующим причинам:

1. **Полнофункциональность**: Встроенные health checks, KV-хранилище, DNS и HTTP API
1. **Интеграция с Traefik**: Нативная поддержка Consul в Traefik как провайдера service discovery
1. **Простота развёртывания**: Легко запускается в Docker Compose для development окружения
1. **Поддержка multi-datacenter**: Возможность расширения для production multi-region развёртывания
1. **Активное сообщество**: Широкая документация и поддержка

Архитектура решения:

- Consul сервер в dev окружении (docker-compose.consul.yml)
- Consul агенты на каждом сервисе для регистрации и health checks
- Traefik настроен как Consul provider для динамической маршрутизации
- Сервисы регистрируются в Consul при запуске через sidecar-контейнеры

## Consequences

### Положительные:

- **Динамическое обнаружение сервисов**: Автоматическая регистрация и обнаружение
- **Health monitoring**: Встроенные health checks для автоматического исключения неработоспособных
  сервисов
- **Упрощение конфигураций**: Устранение хардкода адресов в конфигурациях
- **Подготовка к Kubernetes**: Consul хорошо интегрируется с Kubernetes для production развёртывания
- **Улучшение отказоустойчивости**: Автоматическое переключение на здоровые экземпляры

### Отрицательные:

- **Дополнительная сложность**: Внедрение Consul добавляет новый компонент в инфраструктуру
- **Ресурсы**: Consul требует дополнительных ресурсов CPU и памяти
- **Кривая обучения**: Команде необходимо изучить Consul и его API
- **Миграция**: Существующие сервисы требуют модификации для интеграции с Consul

### Нейтральные:

- **Backward compatibility**: Существующие статические конфигурации остаются работоспособными во
  время миграции
- **Поэтапное внедрение**: Можно внедрять постепенно, сервис за сервисом
