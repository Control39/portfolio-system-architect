# 📚 Руководство: Безопасная работа с Git, LFS и двумя удалёнными репозиториями

> **Внимание!** Это руководство содержит примеры скриптов и конфигураций. Перед использованием:
> * замените плейсхолдеры (`<...>`) на актуальные значения;
> * протестируйте скрипты в тестовом репозитории;
> * убедитесь, что правила LFS и хуков соответствуют вашей инфраструктуре.

**Автор:** Команда разработки  
**Версия:** 1.0 (шаблон)  
**Для:** Разработчиков, использующих Git LFS и несколько удалённых репозиториев  

## 🔐 Цель

Обеспечить:
* безопасное управление секретами;
* чистую историю без LFS‑ошибок;
* надёжную синхронизацию между удалёнными репозиториями;
* автоматизацию рутинных задач.

## 🧩 Структура окружения

* **origin** → `https://<ваш-основной-репозиторий>/<ваш-проект>`;
* **дополнительный remote** → `https://<ваш-зеркало-репозиторий>/<ваш-проект>`;
* **LFS** → включён, но только для нужных файлов;
* **секреты** → `.gigacode/config.ini` — **НИКОГДА** в репозитории.

## 🛑 Запрещено

| Что | Почему |
|------|-------|
| `git add .gigacode/config.ini` | Содержит API‑ключи |
| `git lfs track "*.ini"` | Может зацепить конфиги с секретами |
| Пуш без проверки LFS | Может сломать CI/CD |
| Работа без хуков | Риск случайного коммита секретов |

## ✅ Разрешено

| Что | Как |
|------|-------|
| Хранить шаблон `.gigacode/config.ini.example` | Без ключей |
| Использовать LFS для больших бинарников | `.psd`, `.mp4`, `.pdf` и т. д. |
| Иметь два remote | `origin` — основной, дополнительный — зеркало |

## 🧰 Инструменты

### 1. `git-remove-lfs-file` — удаление файла из истории

**Установка:**
```bash
mkdir -p ~/bin
cat > ~/bin/git-remove-lfs-file << 'EOF'
#!/bin/bash
FILE_PATH="$1"
if [ -z "$FILE_PATH" ]; then
  echo "❌ Ошибка: не указан путь к файлу."
  exit 1
fi
echo "🔍 Удаляем из истории: $FILE_PATH"
# Удаляем файл из всей истории Git
git filter-repo --path "$FILE_PATH" --invert-paths --force || exit 1
echo "✅ Удалено. Выполните:"
echo "   git push origin main --force-with-lease"
echo "   git push <ваш-дополнительный-remote> main --force-with-lease"
EOF
chmod +x ~/bin/git-remove-lfs-file
echo 'export PATH="$HOME/bin:$PATH"' >> ~/.zshrc
```

**Использование:**
```bash
git remove-lfs-file "<путь/к/файлу>"
```

### 2. `pre-commit` — запрет коммита секретов

```bash
cat > .git/hooks/pre-commit << 'EOF'
#!/bin/bash
# Проверяем, не пытаемся ли мы закоммитить файлы из .gigacode/
if git diff --cached --name-only | grep -q "^\.gigacode/"; then
  echo "❌ Запрещено коммитить .gigacode/ — это секреты!"
  exit 1
fi
EOF
chmod +x .git/hooks/pre-commit
```

### 3. `pre-push` — проверка LFS перед пушем

```bash
cat > .git/hooks/pre-push << 'EOF'
#!/bin/bash
echo "🔍 Проверка LFS..."
# Проверяем наличие всех LFS‑файлов
git lfs ls-files | while read hash pointer file; do
  [ ! -f "$file" ] && echo "❌ Отсутствует: $file" && exit 1
done
echo "✅ Все LFS‑файлы на месте"
EOF
chmod +x .git/hooks/pre-push
```

## 🔁 Синхронизация: origin ↔ дополнительный remote

**Вариант 1: Вручную (рекомендуется)**
```bash
git push origin main
git push <ваш-дополнительный-remote> main
```

**Вариант 2: Автоматически (скрипт)**

Создайте `~/bin/git-sync-safe`:
```bash
cat > ~/bin/git-sync-safe << 'EOF'
#!/bin/bash
echo "🔄 Синхронизация: origin → дополнительный remote"
git push origin main --force-with-lease && \
git push <ваш-дополнительный-remote> main --force-with-lease && \
echo "✅ Успешно синхронизировано" || \
echo "❌ Ошибка синхронизации"
EOF
chmod +x ~/bin/git-sync-safe
```

**Использование:**
```bash
git-sync-safe
```

## 🧹 Очистка LFS: пошагово

Если ошибка `GH008: unknown Git LFS object`:
1. Убедитесь, что файл не должен быть в репозитории.
2. Удалите его из истории:
```bash
git remove-lfs-file "<путь/к/файлу>"
```
3. Запушьте:
```bash
git push origin main --force-with-lease
git push <ваш-дополнительный-remote> main --force-with-lease
```

## 🧪 Проверка: что всё в порядке?

```bash
# 1. Проверить, что файл ушёл из истории

# 2. Проверить .gitattributes
cat .gitattributes | grep -v "^#" | grep -q "gigacode" || echo "✅ .gigacode больше не отслеживается"

# 3. Проверить наличие LFS‑файлов
git lfs ls-files
```

## 🤖 GitHub Actions: автоматическая синхронизация (опционально)

Если хотите, чтобы origin автоматически пушил в дополнительный remote:

```yaml
# .github/workflows/sync-to-origin.yml
name: Sync to Main Repo
on:
  push:
    branches: [ main ]
jobs:
  sync:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          lfs: true
      - name: Configure Git
        run: |
          git config user.name "GitHub Action"
          git config user.email "action@github.com"
      - name: Push to origin
        run: |
          git remote add origin-main https://$<ВАШ_ПОЛЬЗОВАТЕЛЬ>:$<ВАШ_ТОКЕН>@<ваш-основной-репозиторий>/<ваш-проект>.git
          git push origin-main main
        env:
          <ВАШ_ПОЛЬЗОВАТЕЛЬ>: ${{ secrets.<ВАШ_ПОЛЬЗОВАТЕЛЬ> }}
          <ВАШ_ТОКЕН>: ${{ secrets.<ВАШ_ТОКЕН> }}
```
> **Требует:** настройки секретов в GitHub Secrets.

## 📌 Best Practices

| Правило | Обоснование |
|--------|-----------|
| Никогда не коммитьте `.env`, `.ini`, `.json` с ключами | Безопасность |
| Используйте `.example` файлы | Для настройки окружения |
| Регулярно проверяйте `.gitattributes` | Чтобы не было лишнего LFS |
| Используйте `pre-commit` и `pre-push` | Защита от ошибок |
| Делайте `fetch` перед работой | Актуальная история |

## Чек‑лист безопасности и команды для проверки


### 1. Проверка на наличие реальных секретов в файлах

**Что проверяем:** нет ли в отслеживаемых файлах API‑ключей, токенов, паролей.


**Команды:**
```bash
# Поиск по ключевым словам в отслеживаемых файлах
git grep -E "(api_key|token|password|secret|credential)"


# Проверка конкретного файла из руководства
git grep -E "(api_key|token)" .gigacode/config.ini.example

# Поиск файлов, которые могут содержать секреты (но не должны быть в репозитории)
git ls-files | grep -E "(.env|\.ini|\.json|\.yaml)" | xargs grep -l -E "(key|token|pass|cred)" || echo "Ничего подозрительного"
```

**Ожидаемый результат:** не должно быть файлов с реальными секретами; в `.gigacode/config.ini.example` — только заглушки (`your_actual_key_here` и т. п.).

### 2. Проверка корректности `.gitignore`


**Что проверяем:** правильно ли настроены правила игнорирования, особенно для секретов и LFS.

**Команды:**
```bash
# Просмотр текущих правил игнорирования
cat .gitignore

# Проверка, игнорируется ли файл с секретами
git check-ignore -v .gigacode/config.ini

# Проверка, не игнорируется ли случайно шаблон конфигурации
git check-ignore -v .gigacode/config.ini.example
```
**Ожидаемый результат:**
* `config.ini` — игнорируется;
* `config.ini.example` — не игнорируется.

### 3. Проверка настроек Git LFS

**Что проверяем:** какие файлы отслеживаются через LFS, нет ли опасных шаблонов (например, `*.ini`).

**Команды:**
```bash
# Список отслеживаемых шаблонов LFS
git lfs track

# Проверка .gitattributes на опасные шаблоны
cat .gitattributes

# Список всех LFS‑файлов в текущей ревизии
git lfs ls-files
```
**Ожидаемый результат:** в отслеживаемых шаблонах нет `*.ini`, `*.json` и т. д.; LFS используется только для бинарников (`.psd`, `.mp4` и т. п.).

### 4. Проверка хуков Git

**Что проверяем:** установлены ли и работают ли хуки `pre-commit` и `pre-push` из руководства.

**Команды:**
```bash
# Проверка наличия хуков
ls -la .git/hooks/ | grep -E "(pre-commit|pre-push)"

# Тестовая проверка хука pre-commit: попытаемся закоммитить файл из .gigacode/
echo "test" > .gigacode/test.txt
git add .gigacode/test.txt
git commit -m "test commit"
# Ожидаем ошибку от хука

# Убираем тестовый файл
git reset HEAD .gigacode/test.txt && rm .gigacode/test.txt
```
**Ожидаемый результат:** хуки существуют и блокируют коммит файлов из `.gigacode/`.

### 5. Проверка удалённых репозиториев

**Что проверяем:** настроены ли два remote (`origin` и дополнительный), корректны ли URL.

**Команды:**
```bash
# Список удалённых репозиториев
git remote -v

# Детальная информация о каждом remote
git remote show origin
```
**Ожидаемый результат:** присутствуют `origin` и дополнительный remote; URL не содержат реальных токенов в открытом виде.


### 6. Проверка истории на наличие секретов

**Что проверяем:** не попали ли случайно секреты в историю ранее.

**Команды:**
```bash
# Поиск в истории коммитов
git log -p --all -G "(api_key|token|secret)" --oneline

# Поиск файлов с потенциально опасными именами в истории
git log --name-only --all | grep -E "(.env|\.ini|\.json)" | sort | uniq
```
**Ожидаемый результат:** нет коммитов, добавляющих файлы с секретами.

### 7. Проверка скрипта `git-remove-lfs-file`

**Что проверяем:** установлен ли скрипт, работает ли.

**Команды:**
```bash
# Проверка доступности скрипта
which git-remove-lfs-file

# Если скрипт не найден, проверяем путь ~/bin/
ls -la ~/bin/git-remove-lfs-file 2>/dev/null || echo "Скрипт не установлен"
```
**Ожидаемый результат:** скрипт доступен в `$PATH` или лежит в `~/bin/`.

### 8. Проверка GitHub Actions (если используется)

**Что проверяем:** в workflow нет жёстко прописанных токенов.

**Команды:**
```bash
# Поиск токенов в YAML‑файлах CI/CD
grep -r "token" .github/workflows/ 2>/dev/null || echo "Токены не найдены в workflow"
```
**Ожидаемый результат:** токены передаются через `secrets`, а не прописаны в коде.

### 9. Общая проверка состояния репозитория

**Команды:**
```bash
# Статус репозитория
git status

# Список игнорируемых файлов (для проверки .gitignore)
git status --ignored

# Проверка подключения к удалённым репозиториям
git fetch --all
```
**Ожидаемый результат:** статус чистый (`nothing to commit`), все remote доступны.

## Итоговый чек‑лист «Готово к коммиту»

Перед тем как делать `git commit`, убедитесь, что:

1. [ ] В отслеживаемых файлах **нет реальных секретов** (проверено `git grep`).
2. [ ] `.gitignore` **правильно игнорирует секреты** (проверено `git check-ignore`).
3. [ ] LFS **не отслеживает конфиги** (`*.ini`, `*.json`) (проверено `git lfs track`).
4. [ ] **Хуки работают** и блокируют опасные коммиты (проверено тестовым коммитом).
5. [ ] Удалённые репозитории **настроены корректно** (проверено `git remote -v`).
6. [ ] История **чиста от секретов** (проверено `git log -G`).
7. [ ] Скрипт `git-remove-lfs-file` **доступен** (проверено `which`).
8. [ ] GitHub Actions **не содержат жёстко прописанных токенов** (проверено `grep`).
9. [ ] Репозиторий **в чистом состоянии** (`git status` — ничего к коммиту).

Если все пункты отмечены — можно коммитить с уверенностью в безопасности!

3. [ ] LFS **не отслеживает конфиги** (проверено `git lfs track`).
4. [ ] **Хуки работают** и блокируют опасные коммиты (проверено тестовым коммитом).
5. [ ] Удалённые репозитории **настроены корректно** (проверено `git remote -v`).
6. [ ] История **чиста от секретов** (проверено `git log -G`).
7. [ ] Скрипт `git-remove-lfs-file` **доступен** (проверено `which`).
8. [ ] GitHub Actions **не содержат жёстко прописанных токенов** (проверено `grep`).
9. [ ] Репозиторий **в чистом состоянии** (`git status` — ничего к коммиту).

Если все пункты отмечены — можно коммитить с уверенностью в безопасности!
```

## Несколько предложений по улучшению:

### 1. Добавить однострочник для быстрой проверки

### 2. Добавить проверку размера LFS-объектов

Иногда проблема не в наличии LFS, а в слишком больших файлах:

```bash
# Проверка размера LFS-файлов
git lfs ls-files -l | awk '{if($3 > 104857600) print "⚠️ Огромный файл:", $0}'

```

### 3. Уточнить про `git-secrets`

Если команда использует AWS или другие сервисы с готовыми инструментами:

```

### 4. Добавить проверку на force push

Force push с LFS может быть опасен:

```bash
# Проверка недавних force push (только если есть доступ к reflog)

```
