# ADR-004: Выбор формата хранения маркеров компетенций

## Статус
Принято

## Контекст
Необходимо выбрать формат хранения маркеров компетенций в IT Compass. Маркеры содержат информацию о навыках, уровнях, критериях выполнения и валидации. Выбор формата влияет на простоту разработки, масштабируемость и возможность интеграции с другими системами.

## Рассмотренные варианты
1. **JSON файлы** - текущая реализация
2. **YAML файлы** - более читаемый формат для людей
3. **Реляционная база данных** (PostgreSQL/MySQL)
4. **NoSQL база данных** (MongoDB)
5. **Графовая база данных** (Neo4j)

## Решение
Выбран формат JSON файлов для хранения маркеров компетенций по следующим причинам:

### Преимущества JSON:
- **Простота**: Не требует установки и настройки СУБД
- **Портативность**: Файлы легко переносятся между системами
- **Человекочитаемость**: Можно редактировать вручную
- **Версионирование**: Отлично работает с Git
- **Интеграция**: Легко интегрируется с LLM и RAG системами
- **Минимализм**: Подходит для текущего масштаба проекта

### Компромиссы:
- **Масштабируемость**: При большом количестве маркеров (>10000) может потребоваться СУБД
- **Сложные запросы**: Для сложных аналитических запросов JSON менее эффективен
- **Целостность данных**: Нет встроенных механизмов обеспечения целостности

## Последствия
### Положительные:
- Простая реализация и поддержка
- Легкая интеграция с Cloud-Reason для анализа
- Поддержка экспорта в форматы, понятные LLM
- Возможность добавления метаданных для RAG-поиска
- Хорошо работает с существующей архитектурой

### Отрицательные:
- Ограничения по масштабируемости при росте объема данных
- Необходимость ручной валидации целостности данных
- Сложность реализации сложных аналитических запросов

## Альтернативы для будущего:
- При росте объема данных (>10000 маркеров) рассмотреть PostgreSQL с JSONB полями
- Для сложных аналитических задач рассмотреть интеграцию с ElasticSearch
