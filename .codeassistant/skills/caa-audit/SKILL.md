# Skill: CAA Audit & Positioning Validator

## Цель
Аудит и валидация позиционирования Cognitive Automation Agent (CAA) как автономной системы, а не просто инструмента. Проверка соответствия между технической реализацией CAA и его бизнес-позиционированием.

## Возможности

### 1. Аудит реализации CAA
- Проверка полноты интеграции с Git (хуки на коммит, пуш, PR)
- Оценка качества автоматических отчётов и диаграмм
- Анализ эффективности мониторинга (количество предотвращённых проблем)
- Измерение скорости реакции на инциденты (оповещения)
- Оценка полезности проактивных действий (автосоздание тестов, исправление багов)

### 2. Валидация позиционирования
- Проверка использования терминологии "автономная система" vs "инструмент"
- Анализ нарратива "Архитектор vs Строитель"
- Оценка доказательной базы (ссылки на артефакты, метрики)
- Проверка целевой аудитории (разработчик, архитектор, руководитель)

### 3. Генерация отчётов
- Структурированные метрики CAA
- Готовые формулировки для разных аудиторий
- Рекомендации по улучшению позиционирования
- Таблицы сравнения "было → стало"

- **Автономность**: 🟢/🟡/🔴 (наличие фраз "предугадывает", "сам создаёт задачи")
- **Интеграция**: 🟢/🟡/🔴 (упоминания Git, CI/CD, Telegram, Prometheus)
- **Нарратив**: 🟢/🟡/🔴 (использование аналогии "Архитектор vs Строитель")
- **Доказательства**: 🟢/🟡/🔴 (ссылки на артефакты, метрики, кейсы)

## Критические правила

### ✅ Обязательные элементы позиционирования CAA:
1. **Автономность**: Должен описываться как "система, которая предугадывает потребности", а не "инструмент, который выполняет команды"
2. **Архитектурный нарратив**: Обязательна аналогия "Архитектор (CAA) vs Строитель (Кодер)"
3. **Интеграция**: Должны быть упомянуты минимум 3 внешние системы (Git, CI/CD, мониторинг, уведомления)
4. **Доказательства**: Каждое утверждение должно иметь ссылку на артефакт (отчёт, лог, диаграмму)

### ❌ Запрещённые формулировки:
- "Простой скрипт/бот"
- "Набор хуков Git"
- "Инструмент автоматизации"
- "CI/CD пайплайн"

## Примеры использования

### Для README:
```markdown
## Cognitive Automation Agent (CAA)

**Автономная система**, которая:
- ✅ **Предугадывает** потребности проекта (анализ контекста, стек технологий)
- ✅ **Создаёт задачи** до их явного запроса (проактивное планирование)
- ✅ **Интегрируется** с экосистемой (Git хуки, CI/CD, мониторинг Prometheus)
- ✅ **Отправляет оповещения** при проблемах (Telegram, Email)

**Архитектор vs Строитель**: CAA работает как архитектор, который проектирует систему, в то время как разработчики (строители) реализуют отдельные компоненты.

**Доказательства**:
- [Отчёт авто-тестов](reports/auto-tests-2024-04-10.md): +15% покрытия за 2 мин
- [Лог оповещений](logs/caa-alerts-2024.log): время реакции 15 мин
- [Конфигурация Git хуков](.agents/config/git-hooks.yaml): 100% проверок коммитов
```

### Для презентации инвесторам:

### Для технического руководителя:
"CAA гарантирует покрытие тестами ≥80% и мгновенно оповещает об уязвимостях. Интеграция с Prometheus и Grafana даёт полную наблюдаемость."

## Интеграция с PMR Agent

### Режим `caa-audit`:
```bash
# Запуск аудита CAA
/pmr --mode=caa-audit

# С фокусом на конкретные аспекты
/pmr --mode=caa-audit --focus=autonomy,integration,narrative

# С генерацией отчёта
/pmr --mode=caa-audit --output=caa-audit-report.md
```

### Проактивные триггеры:
```yaml
# В .agents/config/triggers.yaml
triggers:
  - on_commit:
      paths: ["README.md", "docs/employer/*.md"]
      action: "/pmr --mode=caa-audit --quick --no-save"
  - on_push:
      branches: ["main"]
      action: "/pmr --mode=caa-audit --save"
  - on_schedule:
      cron: "0 9 * * 1"  # Каждый понедельник в 9 утра
      action: "/pmr --mode=caa-audit --full --save"
```

## Метрики успешности

### Количественные:
- **Score позиционирования**: 0-100 (рассчитывается на основе критериев)
- **Время реакции**: от обнаружения проблемы до оповещения
- **Экономия времени**: часы/неделя, сэкономленные на рутине
- **Coverage улучшение**: % увеличения покрытия тестами

### Качественные:
- **Убедительность для HR**: низкая/средняя/высокая
- **Убедительность для инвесторов**: низкая/средняя/высокая
- **Техническая глубина**: поверхностная/умеренная/глубокая

## Шаблоны ответов на возражения

### Возражение 1: "Это просто набор хуков Git"
**Ответ**: "CAA не просто запускает скрипты — он анализирует проект, находит проблемы и предлагает решения. Например, после коммита он не только запускает тесты, но и создаёт отчёт о покрытии и диаграмму связей сервисов."
**Доказательство**: `docs/cases/caa-git-hooks.md`

### Возражение 2: "А если агент ошибётся и сломает проект?"
**Ответ**: "CAA работает в режиме read‑only по умолчанию. Все изменения предлагаются как PR с подробным описанием. Перед применением проводится проверка безопасности."
**Доказательство**: `docs/security/caa-safety-protocol.md`

### Возражение 3: "Зачем это бизнесу?"
**Ответ**: "CAA заменяет рутинный контроль архитектора. Он находит протечки в фундаменте проекта до того, как они станут багами, экономя часы на ревью и тысячи на исправления в production."
**Доказательство**: `reports/caa-roi-2024.md`
