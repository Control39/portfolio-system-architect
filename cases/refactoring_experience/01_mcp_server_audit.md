# Аудит и рефакторинг MCP-Server после миграции структуры

**Дата аудита:** 5 мая 2026 г.  
**Микросервис:** `apps/mcp-server`  
**Ответственный:** Екатерина Куделя, Cognitive Systems Architect  
**Статус:** Выявлены критические проблемы, требуется рефакторинг

---

## 🎯 Проблема

После глобальной реорганизации репозитория (миграция из корня в структуру `apps/`) микросервис `mcp-server` потерял целостность импортов и нарушил паттерны инициализации. Это привело к:

- **Нарушению зависимостей** — импорты указывают на несуществующие пути
- **Отсутствию критических модулей** — инструменты навигации и команд не найдены
- **Архитектурному конфликту** — дублирование экземпляров `FastMCP` в одном процессе

Проблема выявлена в ходе глубокого аудита перед интеграцией в CI/CD pipeline.

---

## 🔍 Диагностика

### Критические проблемы (5 найдено)

| № | Тип проблемы | Описание | Серьёзность |
|---|--------------|----------|-------------|
| 1 | **ModuleNotFoundError** | `ModuleNotFoundError: No module named 'navigation'` | 🔴 Критическая |
| 2 | **ModuleNotFoundError** | `ModuleNotFoundError: No module named 'command_tools'` | 🔴 Критическая |
| 3 | **ModuleNotFoundError** | `ModuleNotFoundError: No module named 'src.shared.llm'` | 🔴 Критическая |
| 4 | **Архитектурная ошибка** | Дублирование инициализации `FastMCP()` в `main.py` (строки 20 и 35) | 🟠 Высокая |
| 5 | **Зависимости** | В `requirements.txt` отсутствуют явные зависимости для `src.shared` | 🟡 Средняя |

### Отсутствующие файлы

```
❌ apps/mcp-server/navigation.py        — модуль навигации по файловой системе
❌ apps/mcp-server/command_tools.py     — модуль выполнения системных команд
❌ src/shared/llm/__init__.py           — инициализация общего LLM-слоя (проверка требуется)
```

### Дополнительная информация

- **sys.path manipulation**: В коде отсутствуют явные манипуляции с `sys.path`, что приводит к ошибкам импорта при запуске из разных директорий.
- **Паттерн инициализации**: Отсутствует единый стандарт для функций `init_*`, что усложняет тестирование и поддержку.

---

## 🏗️ Архитектурное решение

### План исправлений

1. **Унификация паттернов инициализации**
   - Создать единый паттерн `init_tools()`, `init_llm()`, `init_mcp()`
   - Вынести инициализацию в отдельные функции с явными зависимостями
   - Устранить дублирование экземпляров `FastMCP`

2. **Очистка и актуализация зависимостей**
   - Добавить явные зависимости в `requirements.txt`:
     ```txt
     src-shared>=0.1.0
     navigation-tools>=0.1.0
     command-tools>=0.1.0
     ```
   - Или использовать относительные импорты внутри `apps/mcp-server/`

3. **Стабилизация `sys.path`**
   - Добавить явное управление `sys.path` для корректного импорта:
     ```python
     import sys
     from pathlib import Path
     sys.path.insert(0, str(Path(__file__).parent.parent.parent))
     ```

4. **Восстановление отсутствующих модулей**
   - Создать заглушки `navigation.py` и `command_tools.py` с базовой функциональностью
   - Или переписать импорты на использование существующих модулей из `tools/` или `src/`

5. **Добавление тестов**
   - Создать `tests/test_mcp_server_init.py` для проверки инициализации
   - Добавить интеграционные тесты для импортов

## 📊 Выводы

### Ключевые уроки

1. **Единые стандарты критически важны**  
   При 15 микросервисах отсутствие единого паттерна импортов и инициализации приводит к каскадным ошибкам при миграции.

2. **Аудит перед интеграцией необходим**  
   Глубокий статический анализ до включения в CI/CD позволяет выявить проблемы на ранней стадии.

3. **Явные зависимости > неявных импортов**  
   Использование `sys.path` и относительных импортов должно быть документировано и стандартизировано.

4. **Автоматизация проверки**  
   Требуется добавить pre-commit hook для проверки импортов во всех микросервисах.
