# Runbook: Процесс реагирования на инциденты

> **Уровень критичности:** CRITICAL
> **Время реакции (целевое):** <5 минут
> **Автор:** Ekaterina Kudelya
> **Обновлено:** 18 мая 2026

---

## 🚨 Типы инцидентов

| Уровень | Описание | Пример | Время реакции |
|---------|----------|--------|---------------|
| **P0 - Critical** | Полный простой системы | Все сервисы упали | <5 мин |
| **P1 - High** | Критический сервис недоступен | Auth, DB, API Gateway | <15 мин |
| **P2 - Medium** | Частичная деградация | Один из 15 сервисов | <1 час |
| **P3 - Low** | Неблокирующая проблема | Логирование, метрики | <24 час |

---

## 🔄 Процесс реагирования

### Шаг 1: Обнаружение (0-2 мин)

**Автоматическое:**
- AlertManager отправляет notification
- Telegram: @ops-alerts
- Email: leadarchitect@yandex.ru

**Ручное:**
- Пользователь сообщил
- Мониторинг показал аномалию

---

### Шаг 2: Оценка (2-5 мин)

```bash
# Быстрая проверка
.\navigate.ps1 -Status

# Проверка критических сервисов
curl http://localhost:8100/health  # Auth
curl http://localhost:5432         # PostgreSQL
curl http://localhost:8080/health  # API Gateway
```

**Определить уровень:**
- P0: Все сервисы недоступны → Эскалация CTO
- P1: Критический сервис → On-call немедленно
- P2: Один сервис → Обычный процесс
- P3: Незначительное → В очередь задач

---

### Шаг 3: Митигация (5-15 мин)

**Приоритет 1: Восстановить сервис**
- Перезапуск (см. `service-down.md`)
- Откат версии (см. `deployment-rollback.md`)
- Бэкап восстановления (см. `db-recovery.md`)

**НЕ тратить время на:**
- Поиск первопричины (после восстановления)
- Детальный анализ логов
- Обсуждение вины

---

### Шаг 4: Восстановление (15-30 мин)

```bash
# Проверить все сервисы
docker-compose ps

# Проверить метрики
open http://localhost:3000

# Проверить логи ошибок
docker-compose logs --tail=50 | grep ERROR
```

---

### Шаг 5: Пост-мортем (24 часа)

Создать issue с:

```markdown
## Инцидент: <краткое описание>

**Время:** 2026-05-18 14:30 - 14:45 UTC
**Уровень:** P1
**Сервис:** auth-service
**Длительность:** 15 минут

### Что произошло
Пользователи не могли авторизоваться

### Причина
Database connection pool исчерпан

### Действия
- Перезапуск auth-service (не помогло)
- Проверка PostgreSQL (перегруз)
- Увеличение pool size (успешно)

### Предотвращение
- [ ] Добавить алерты на connection pool
- [ ] Увеличить max_connections
- [ ] Добавить circuit breaker
```
