    Кейс: Миграция cloud_reason → decision_engine — восстановление ядра и устранение техдолга

    Статус: ✅ Завершено Дата: Май 2026 Роль: Cognitive Systems Architect Репозиторий: portfolio-system-architect Связанные компоненты: src/decision_engine/, apps/decision-engine/, old-repo-check

    🎯 Контекст и проблема

    Исходная ситуация

    • Компонент cloud_reason (оркестрация ИИ, API, утилиты) был переименован в decision_engine в ходе рефакторинга.
    • При миграции возникла двойная вложенность: src/decision_engine/decision_engine/.
    • Агенты, анализируя структуру, ошибочно помечали проект как «прототип» из-за:
    • Потерянных утилит (error_handler, rate_limiter, session_store)
    • Отсутствия интеграционных тестов
    • Устаревших импортов (from src.decision_engine.decision_engine import ...)
    • Дублирования логики в .py/.sh скриптах

    Бизнес-риск

    • ❌ Тесты падали с ModuleNotFoundError
    • ❌ Агенты и ревьюеры не видели целостной архитектуры
    • ❌ Угроза для заявки на SourceCraft: «незавершённый проект»

    🔍 Расследование (Root Cause Analysis)

    Шаг 1: Аудит истории

    # Поиск удалений в src/
git log --oneline --diff-filter=D -- src/

# Анализ конкретных коммитов
git show 7f860ad1 --stat  # Удаление дубликатов-копий (безопасно)
git show f34d1377 --stat  # Ошибочное удаление ядра (проблема)

    Шаг 2: Сравнение с архивом

    # Приватный репо как эталон
diff -rq ~/old-repo-check/src/ ~/portfolio-system-architect/src/

# Поиск "потерянных" файлов
find ~/old-repo-check/apps/cloud-reason/ -name "*.py" | grep -v __pycache__

    Шаг 3: Карта миграции

    Компонент Старый путь Новый путь Статус
    Ядро (main.py, api/) apps/cloud-reason/cloud_reason/ src/decision_engine/ ✅ Перенесено
    Утилиты cloud_reason/utils/{error_handler,rate_limiter,session_store}.py ❌ Утеряны 🔴 Восстановлено
    Тесты apps/cloud-reason/tests/ ❌ Утеряны 🔴 Восстановлено
    Импорт в тестах src.decision_engine.decision_engine src.decision_engine ✅ Исправлено

    🛠️ Решение

    1. Восстановление структуры

    # Схлопывание двойной вложенности
mv src/decision_engine/decision_engine/* src/decision_engine/
rmdir src/decision_engine/decision_engine/

# Восстановление утилит из архива
cp ~/old-repo-check/apps/cloud-reason/cloud_reason/utils/{error_handler,rate_limiter,session_store}.py \
   src/decision_engine/utils/

# Восстановление интеграционных тестов
mkdir -p src/decision_engine/tests/
cp ~/old-repo-check/apps/cloud-reason/tests/*.py src/decision_engine/tests/

    2. Фикс импортов

    # Обновление путей в тестах
sed -i 's/cloud_reason/decision_engine/g' src/decision_engine/tests/test_*.py

# Проверка: старых путей больше нет
grep -r "cloud_reason" src/decision_engine/tests/  # Тишина = успех

    3. Устранение дубликатов

    # Удаление .sh-обёртки (логика полностью в Python)
git rm scripts/update-coverage-badge.sh

# Обновление документации
sed -i 's|update-coverage-badge.sh|update-coverage-badge.py|g' docs/*.md .kodacli/KODA.md

    4. Исправление линтера

    # scripts/update-coverage-badge.py: удаление неиспользуемой переменной
# Было:
history_pattern = r"(\| \d{4}-\d{2}-\d{2} \| \d+% \|)"  # F841

# Стало: (удалено)

# Разбиение длинной строки (E501)
new_entry = (
    f"| {datetime.now().strftime('%Y-%m-%d')} | {coverage}% | "
    "Автоматическое обновление через скрипт |"
)

    5. Фиксация правил для ИИ

    Создан .ai-rules.md:

    ## 🧹 Правила очистки для AI-агентов
- `src/` = общие библиотеки. Не удалять директории с `main.py`, `api/`, `core/`.
- Дубликаты с суффиксами `(2)`, `-1` — можно удалять.
- Спорное → перемещать в `.archive/`, а не удалять.
- Перед удалением >50 строк: `git diff --stat` + запрос подтверждения.

    💡 Извлечённые уроки

    Для инфраструктуры

    1. Архив — это страховка. Приватный old-repo-check позволил восстановить утраченное без потерь.
    2. Границы папок должны быть явными. .ai-rules.md предотвращает повторение ошибок агентов.
    3. Один источник истины. Дублирование .py/.sh создаёт риск рассинхрона — лучше оставить один вариант.

    Для коммуникации

    «Проект прошёл эволюцию: монолитное ядро из cloud_reason декомпозировано в decision_engine как переиспользуемый модуль. В src/ остались только общие библиотеки, бизнес-логика — в apps/. Архитектурная миграция ≠ прототип.»

    Эта формулировка закрывает вопрос «почему структура изменилась» и демонстрирует осознанное управление жизненным циклом.

    Для будущих миграций

    # Чек-лист перед переименованием компонента:
1. [ ] Найти все импорты: grep -r "old_name" . --include="*.py"
2. [ ] Обновить тесты и доки ДО удаления старой папки
3. [ ] Сделать коммит-снэпшот: git commit -m "snapshot before rename"
4. [ ] Переименовать → протестировать → закоммитить
5. [ ] Удалить старую папку только после успешных тестов
