# План: Развитие архитектуры портфолио-системы

## Цель
Усилить портфолио-проект за счет внедрения продвинутых архитектурных паттернов: Event-Driven Architecture и Self-Healing Systems. Основной фокус — демонстрация навыков системного архитектора Senior-уровня.

## Контекст
Проект представляет собой комплекс из 14 микросервисов. Текущая архитектура демонстрирует широкий спектр технологий, но требует улучшения в части взаимодействия сервисов и автономности.

## Задачи
1.  **Консолидация утилит.** Убедиться, что `Template Service` и `Career Dev` не создают избыточной нагрузки.
2.  **Внедрение Event Bus.** Реализовать асинхронное взаимодействие между сервисами.
3.  **Разработка AI DevOps Agent.** Создать сервис для автономного мониторинга и восстановления системы.

---

## Детальный план

### 1. Консолидация утилит
- **Цель:** Упростить архитектуру, сохранив демонстрационную ценность.
- **Действия:**
  - Проверить код `apps/template-service` и `apps/career-dev`.
  - Убедиться, что они не содержат тяжелой бизнес-логики или баз данных.
  - При необходимости, задокументировать их как легковесные утилиты.
- **Критические файлы:**
  - `apps/template-service/main.py`
  - `apps/career-dev/main.py`

### 2. Внедрение Event Bus
- **Цель:** Заменить прямые REST-вызовы между сервисами на асинхронную шину событий для демонстрации паттерна Event-Driven Architecture.
- **Технология:** Использовать Redis в качестве брокера сообщений (проще в настройке, чем RabbitMQ/Kafka, и уже может быть в проекте).
- **Действия:**
  - Добавить конфигурацию Redis в `docker-compose.yml`.
  - Обновить `pyproject.toml` или `requirements.txt`, добавив `redis`.
  - Модифицировать `Job Automation Agent` для публикации событий (например, `job_found`) в Redis.
  - Модифицировать `Portfolio Organizer` для подписки на события и обработки новых вакансий.
  - Обновить `ARCHITECTURE.md`, добавив описание шины событий.
- **Критические файлы:**
  - `docker-compose.yml`
  - `requirements.txt` (или `pyproject.toml`)
  - `apps/job-automation-agent/main.py`
  - `apps/portfolio-organizer/main.py`
  - `ARCHITECTURE.md`

### 3. Разработка AI DevOps Agent
- **Цель:** Создать сервис, демонстрирующий концепцию Self-Healing Systems и автономной DevOps-автоматизации.
- **Функционал:**
  - Мониторинг логов сервисов (из папки `logs/`).
  - Обнаружение критических ошибок (например, `ERROR`, `CRITICAL`, `Exception`).
  - Взаимодействие с `Infra Orchestrator` для перезапуска упавших сервисов.
  - (Опционально) Анализ метрик и предложение оптимизаций.
- **Действия:**
  - Создать новый сервис в `apps/devops-agent/`.
  - Реализовать логику чтения лог-файлов и парсинга ошибок.
  - Реализовать REST-клиент для взаимодействия с `Infra Orchestrator`.
  - Написать основной цикл агента (polling логов).
  - Добавить сервис в `docker-compose.yml`.
  - Документировать архитектуру агента в `ARCHITECTURE.md`.
- **Критические файлы:**
  - `apps/devops-agent/main.py` (новый)
  - `docker-compose.yml`
  - `ARCHITECTURE.md`

## Проверка (Verification)
1.  Запустить всю систему с помощью `docker-compose up`.
2.  Убедиться, что `devops-agent` запускается без ошибок.
3.  Сымитировать сбой `Job Automation Agent` (остановить контейнер вручную).
4.  Проверить, что `devops-agent` обнаружил ошибку в логах и отправил команду на перезапуск через `Infra Orchestrator`.
5.  Убедиться, что `Job Automation Agent` был успешно перезапущен.
6.  Проверить, что события о найденных вакансиях корректно передаются через Redis и обрабатываются `Portfolio Organizer`.
