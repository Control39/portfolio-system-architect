    🎯 mcp_server: Покрытие тестами

    Текущий статус

    • Покрытие: 54.12%
    • Всего тестов: 151 (146 passed, 5 failed - pre-existing async issues)
    • Целевой показатель: 80%
    • Статус: Отложено до v1.1 (низкий приоритет)
    • Решение задокументировано в: ADR-017

    Обоснование решения

    Подробное обоснование, альтернативы и последствия описаны в ADR-017.

    Кратко

    • Проблема: Инструменты обернуты в MCP-декораторы, требующие сложных моков
    • Анализ: Следующие 26% покрытия потребуют 80% дополнительных усилий
    • Решение: Оставить 54% для v1.0.0-rc1, рефакторинг в v1.1
    • ROI: 1% покрытия = 10 минут (бизнес-логика) vs 90 минут (моки декораторов)

    План завершения (v1.1)

    Вариант A: Рефакторинг архитектуры (рекомендуется)

    Цель: Упростить тестирование инструментов

    Задачи: 1. Вынести бизнес-логику из-под декораторов 2. Создать чистые функции для тестирования 3. Обернуть в MCP-декораторы только на уровне API

    Оценка: 6-8 часов Риск: Средний (требует изменений в production коде)

    Вариант B: Интеграционные тесты

    Цель: Написать полные моки для MCP-сервера

    Задачи: 1. Создать MockFastMCP class 2. Тестировать инструменты через mock-сервер 3. Добавить 100+ интеграционных тестов

    Оценка: 4-6 часов Риск: Высокий (хрупкие тесты, ломаются при изменении MCP API)

    Вариант C: Принять 54% как достаточное

    Цель: Сфокусироваться на других компонентах

    Обоснование: - 11 из 15 сервисов имеют ≥80% - mcp_server работает стабильно (146 тестов) - Критичная бизнес-логика покрыта

    Решение: Зафиксировать как технический долг, вернуться при необходимости

    Как подать на собеседовании

    «В модуле mcp_server инструменты обернуты в кастомные MCP-декораторы, что усложняет юнит-тестирование. Я оценила затраты: достичь 80% покрытия потребовало бы 3-4 часов на написание хрупких моков.

    Приняла решение выпустить релиз с 54% покрытием (146 тестов), зафиксировав технический долг в документации. Это позволило сфокусироваться на критичных для production компонентах (persistence-слой, E2E).

    Покрытие тестами — инструмент, а не цель. Важнее демонстрировать системное мышление и умение принимать взвешенные решения на основе данных.»
