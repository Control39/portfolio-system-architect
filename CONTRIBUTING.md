# Руководство для контрибьюторов

Спасибо за интерес к проекту `portfolio-system-architect`!

---

## Принципы проекта

1. **Системность** — все компоненты должны быть частью единой экосистемы
2. **Практичность** — каждый компонент должен решать реальную задачу
3. **Масштабируемость** — архитектура должна поддерживать расширение
4. **Документированность** — каждый компонент должен иметь полную документацию

---

## Приоритетные направления вклада

Мы особенно приветствуем следующие типы вклада:

1. **Новые кейсы** — добавление практических примеров системного мышления
2. **Оптимизация компонентов** — улучение производительности и качества кода
3. **Улучшение документации** — исправление ошибок, добавление примеров
4. **Новые компоненты** — расширение экосистемы новыми инструментами
5. **Тесты** — повышение покрытия тестами

---

## Как внести вклад

### 1. Форкните репозиторий

Создайте форк на GitHub.

### 2. Клонируйте

```bash
git clone https://github.com/Control39/cognitive-systems-architecture.git
cd portfolio-system-architect
```

### 3. Создайте ветку

```bash
git checkout -b feature/название-функции
```

### 4. Внесите изменения

- Исправьте ошибку в коде или документации
- Добавьте новый компонент в `components/`
- Создайте кейс в `cases/`
- Добавьте пример использования в `examples/`

### 5. Sign-off & Commit (DCO)

**Developer Certificate of Origin (DCO)**: Certify your work is original.

Use `git commit -s` (auto-adds Signed-off-by):

```bash
git commit -s -m "feat: brief change description

Detailed body...

Signed-off-by: Your Name <email@example.com>"
```

**DCO Template**:
```
feat/module: description

Body...

Signed-off-by: Name <email>
```

### 6. Push & Create PR

```bash
git push origin feature/название-функции
```

Создайте Pull Request с описанием изменений.

### Перед созданием PR убедитесь, что:

- [ ] Ветка обновлена до последней версии `main` (`git pull origin main`)
- [ ] Все тесты проходят (`python -m pytest tests/`)
- [ ] Код соответствует стилю проекта (`flake8 .`, `black .`)
- [ ] Дубликаты проверены (`python scripts/duplicate_finder.py`)
- [ ] Документация обновлена при необходимости

## Процесс ревью и мерджа

### Current: Solo Architect Lab

1. **Перед мерджем в `main` выполните саморевью по чек‑листу**:
   - Проверьте работоспособность изменений
   - Убедитесь в соблюдении стандартов кода
   - Обновите документацию при необходимости
   - Удалите отладочный код
   - Проверьте сообщения коммитов

2. **В описании PR кратко укажите**:
   - Цель изменений
   - Затронутые компоненты
   - Инструкции по проверке (если применимо)

3. **После саморевью выполните мердж в `main`**

### Будущие правила (при появлении контрибьюторов)

1. **Для PR от внешних контрибьюторов**:
   - Требуется одобрение минимум одного ревьюера
   - Срок рассмотрения — до 3 рабочих дней
   - При спорных вопросах решение принимает ведущий разработчик

2. **Для изменений ведущего разработчика**:
   - Саморевью по чек‑листу обязательно
   - Внешний ревьюер желателен для сложных изменений

### Критерии «сложных изменений» (требуют внешнего ревью при возможности)

- Затрагивают несколько компонентов
- Меняют архитектуру проекта
- Вносят критические исправления безопасности
- Требуют значительных ресурсов на тестирование

## Code Review Guidelines

### Общие принципы Code Review

1. **Цель ревью** — улучшение качества кода, обмен знаниями и поддержание стандартов проекта
2. **Конструктивная критика** — фокусируйтесь на коде, а не на авторе
3. **Своевременность** — старайтесь провести ревью в течение 24-48 часов
4. **Обучение** — используйте ревью как возможность для обучения и передачи знаний

### Стандарты кода для ревью

#### Python
- Соответствие PEP 8 и внутренним стандартам проекта
- Наличие type hints для всех публичных функций
- Документация (docstrings) для модулей, классов и функций
- Обработка ошибок и логирование
- Тестовое покрытие новых функций

#### PowerShell
- Следование PowerShell Best Practices
- Использование строгой типизации (`[string]`, `[int]` и т.д.)
- Комментарии для сложных логических блоков
- Проверка на null и обработка исключений

#### Документация
- Обновление README.md при изменении функциональности
- Добавление примеров использования для новых API
- Обновление схем архитектуры при изменении структуры

### Процесс Code Review

1. **Подготовка к ревью**:
   - Убедитесь, что CI/CD pipeline прошел успешно
   - Проверьте, что тесты покрывают новые изменения
   - Убедитесь в отсутствии конфликтов с основной веткой

2. **Проведение ревью**:
   - Проверьте соответствие стандартам кода
   - Оцените архитектурные решения
   - Проверьте безопасность (нет ли утечек чувствительных данных)
   - Убедитесь в наличии тестов для новой функциональности

3. **Комментарии и исправления**:
   - Используйте конкретные, actionable комментарии
   - Предлагайте альтернативные решения, если есть лучшие подходы
   - Отмечайте позитивные аспекты изменений

4. **После ревью**:
   - Автор вносит исправления на основе комментариев
   - Повторное ревью для проверки исправлений
   - Approval от минимум одного ревьюера для внешних контрибьюторов

### Автоматизированные проверки

Перед ручным ревью убедитесь, что пройдены все автоматические проверки:

- ✅ Линтинг (ruff, black, flake8)
- ✅ Тесты (pytest с покрытием >80%)
- ✅ Проверка безопасности (trivy, safety)
- ✅ Проверка дубликатов кода
- ✅ Сборка Docker образов

## Требования к коду

- Пишите docstrings
- Обрабатывайте ошибки с помощью исключений
- Используйте логирование для отладки

```python
import logging

logger = logging.getLogger(__name__)

def function(param: str) -> dict:
    """Краткое описание.
    
    Args:
        param: Описание параметра.
    
    Returns:
        Описание возвращаемого значения.
    
    Raises:
        ValueError: Описание условия ошибки.
    """
    try:
        result = process(param)
        logger.info(f"Successfully processed: {param}")
        return result
    except Exception as e:
        logger.error(f"Error processing {param}: {e}")
        raise
```

## Документация

Каждый новый компонент должен включать:

- Описание в `README.md`

- **Ссылки**: используйте относительные пути для внутренних ссылок

### Полезные ресурсы

- [Markdown Guide](https://www.markdownguide.org/)

## Правила именования файлов и папок

### Основные принципы
1. **Лаконичность и ясность** — имена должны отражать содержимое
2. **Единый стиль** — используйте kebab-case для папок и файлов
3. **Английский язык** — все имена файлов и папок на английском

### Специфичные правила
- **Папки компонентов**: `component-name/` (например: `it-compass/`, `portfolio-organizer/`)
- **Папки кейсов**: `case-category/case-name/` (например: `thinking-cases/01-ai-communication-breakthrough/`)
- **Файлы документации**: `topic-name.md` (например: `architecture-tradeoffs.md`)
- **Скрипты**: `action-purpose.extension` (например: `duplicate_finder.py`, `generate_docs.py`)

### Примеры правильных и неправильных имён

| ✅ Правильно | ❌ Неправильно |
|-------------|---------------|
| `it-compass/` | `ITCompass/` |
| `duplicate-finder.py` | `duplicate finder.py` |
| `architecture-tradeoffs.md` | `architecture tradeoffs.md` |
| `case-01-analysis/` | `Case 01 Analysis/` |

### Запрещённые имена
- Общие имена: `temp/`, `test/`, `new/`
- Пробелы в именах файлов и папок
- Специальные символы кроме `-`, `_`, `.`

## Предотвращение дублирования контента

### Принципы
1. **Единственный источник истины** — каждый фрагмент информации должен храниться в одном месте
2. **Ссылки вместо копирования** — используйте относительные ссылки между документами
3. **Регулярный аудит** — ежемесячная проверка на дубликаты

### Скрипт duplicate_finder.py

Скрипт ищет дубликаты по:
- **Хешу содержимого** — для бинарных файлов
- **Текстовому сходству** — для `.md`, `.py`, `.yaml` и т.д.

Он игнорирует:
- Временные файлы (указанные в `.gitignore`)
- Примеры в `examples/` (допустимое дублирование для обучения)

### Использование

```bash
# Проверка на дубликаты
python scripts/duplicate_finder.py

# CI-режим (для автоматических проверок)
python scripts/duplicate_finder.py --ci-mode
```

### Порядок разрешения конфликтов

1. Определите каноническую версию (более полная, актуальная)
2. Удалите дубликат
3. Обновите все ссылки на каноническую версию
4. Если версии равноценны — выберите ту, что ближе к корневой структуре

### Исключения

Дублирование допустимо в случаях:
- Примеры в `examples/` — для образовательных целей
- Зеркала для совместимости (требуется документация)
- Временные файлы до рефакторинга

## CI/CD Пайплайн

### Полный пайплайн

```yaml
name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.9'
      - name: Install dependencies
        run: |
          pip install -r requirements.txt
          pip install -r requirements-dev.txt
      - name: Run tests
        run: pytest --cov=. tests/
      - name: Check style
        run: |
          flake8 .
          black --check .

  duplicates:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Check duplicates
        run: python scripts/duplicate_finder.py --ci-mode

  docs:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Build documentation
        run: python scripts/generate_docs.py
```

### Где смотреть результаты

- **Логи**: вкладка "Actions" на GitHub
- **Артефакты**: секция "Artifacts" в результатах запуска
- **Покрытие**: отчёт в интерфейсе GitHub

### Восстановление после сбоев

1. Проверьте логи ошибок
2. Исправьте проблему локально
3. Обновите ветку: `git fetch origin && git rebase origin/main`
4. Отправьте исправления: `git push origin feature/ваша-ветка`
