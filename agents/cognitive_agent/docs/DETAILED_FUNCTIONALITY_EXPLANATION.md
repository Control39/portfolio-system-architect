# Подробное объяснение функциональности Cognitive Agent

## 1. Сканирование проекта

### Обработка .gitignore через pathspec

Агент использует библиотеку `pathspec` для обработки файлов `.gitignore`:

- Файлы, указанные в `.gitignore`, **автоматически исключаются** из сканирования
- Это уважает намерения разработчиков, которые явно указали, что эти файлы не должны быть включены в анализ
- Однако агент **не имеет встроенного механизма** для "включения" файлов из `.gitignore` обратно для специфических задач
- Для таких случаев пользователю нужно временно изменить `.gitignore` или использовать режим `paths` с явным указанием файлов

### Типы обнаруженных проблем

Текущие лимиты и настройки:

- **Большие файлы**: >10 МБ (жестко задано в коде как `10 * 1024 * 1024` байт)
- **Слишком много файлов**: >1000 файлов в директории (жестко задано в коде)
- **TODO-комментарии**: Обнаруживаются через регулярные выражения
- Эти лимиты **не настраиваемы** через конфигурационные файлы, они зашиты в коде

### Режимы сканирования

- **auto**: Сначала проверяет изменения через Git (`git status` или `git diff --name-only HEAD^ HEAD`), затем сканирует только измененные файлы
- **git_diff**: Сканирует только файлы, измененные в последнем коммите или между ветками
- **full**: Сканирует все файлы в проекте
- **paths**: Сканирует только указанные пути файлов

Агент определяет изменения через вызовы Git команд, такие как `git status` и `git diff`, чтобы идентифицировать измененные файлы.

### Использование hashlib

MD5-хэши используются для:

- **Контроля изменений файлов** между сканированиями
- **Дедупликации** при сканировании (чтобы не обрабатывать один и тот же файл дважды)
- **Кэширования результатов** сканирования

## 2. Зависимости между сервисами

### Анализ импорт-зависимостей

- **Standard версия**: Не содержит прямого парсинга импорт-зависимостей между модулями
- **Enterprise версия**: Имеет `TaskPlanner` с графом зависимостей **задач**, а не кода напрямую
- **IT Compass**: Обнаруживает архитектурные артефакты, но не анализирует импорты

### Обнаружение новых зависимостей

Когда добавляется импорт `from old_service.utils import some_function` в новом сервисе:

- Агент **не обнаруживает** эту зависимость автоматически через статический анализ кода
- Он может выявить косвенные признаки зависимости через:
  - Анализ Git-истории на предмет изменений, затрагивающих оба сервиса
  - Поиск совместных изменений в коммитах
  - Анализ логов выполнения задач (если сервисы работают вместе)

### RBAC и управление зависимостями

EnterpriseGuardrails с RBAC позволяют:

- Ограничивать права на изменение критических сервисов только определенными ролями
- Требовать одобрения для изменений в сервисах, от которых зависят другие
- Контролировать доступ к API и внутренним интерфейсам

## 3. Система "Skills"

### Формирование плана выполнения задачи

Когда поступает задача "рефакторить функцию X", агент выполняет следующую последовательность:

1. **Анализ задачи**: Определение типа задачи (рефакторинг, оптимизация и т.д.)
2. **Сканирование контекста**: Использование `ProjectScanner` для поиска функции X
3. **Сбор информации**: Извлечение кода функции и связанного контекста
4. **Генерация плана**: Использование AI-провайдера для создания последовательности действий
5. **Валидация плана**: Применение `guardrails` к каждому шагу
6. **Выполнение**: Пошаговое выполнение с постоянной проверкой безопасности
7. **Проверка результата**: Повторная валидация результата через `guardrails`

### Взаимодействие guardrails с AI-генерацией

- **Предотвращение**: Guardrails блокируют потенциально опасные действия до выполнения
- **Фильтрация**: AI-провайдер не получает прямого доступа к файловой системе без проверки
- **Ограничение**: Доступ к системным командам и чувствительным операциям ограничен
- **AI не знает о guardrails напрямую**, но генерирует код в соответствии с контекстом, который учитывает безопасность

## 4. RAG и ChromaDB

### Индексируемые типы документов

Агент индексирует следующие типы файлов:

- **Кодовые файлы**: `.py`, `.js`, `.ts`, `.jsx`, `.tsx`, `.java`, `.cpp`, `.go`, `.rs` и другие
- **Конфигурационные файлы**: `.json`, `.yaml`, `.yml`, `.toml`, `.ini`, `.cfg`
- **Документационные файлы**: `.md`, `.rst`, `.txt`
- **HTML/CSS файлы**: `.html`, `.css`, `.scss`, `.less`
- **SQL файлы**: `.sql`

Все индексируемые типы файлов определены в конфигурации сканера, обычно в `config/scanner-config.json`.

### Метод search_similar_documents()

Метод позволяет выполнять семантический поиск по:

- Содержимому файлов
- Комментариям и документации
- Структуре кода (через преобразование в embeddings)
- Контекстуальной информации из архитектурных артефактов
