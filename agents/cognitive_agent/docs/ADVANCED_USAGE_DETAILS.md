# Подробное объяснение продвинутых аспектов использования Cognitive Agent

## 1. Стратегия выбора AI-моделей

### Классификация задач

Агент определяет тип задачи через анализ ключевых слов и контекста:

1. **Повседневные задачи** (gigachat-lite):
   - Ключевые слова: "рефакторинг", "багфикс", "форматирование", "тесты"
   - Простые операции с кодом
   - Задачи с четким описанием и ожидаемым результатом

2. **Сложные архитектурные задачи** (gigachat-pro):
   - Ключевые слова: "архитектура", "дизайн", "паттерн", "рефакторинг системы"
   - Анализ зависимостей и проектирование
   - Задачи, требующие глубокого понимания контекста

3. **Offline-режим** (ollama-qwen2.5-coder):
   - Недоступность облачных провайдеров
   - Задачи с чувствительными данными
   - Быстрые правки и локальная разработка

### Критерии переключения на YandexGPT

Агент переключается на YandexGPT как fallback, когда:

1. Все основные модели (GigaChat, Ollama) недоступны
2. Превышено количество попыток подключения
3. Получена ошибка аутентификации или квоты для всех основных моделей

## 2. Детали RBAC и Enterprise Guardrails

### Реализация ролей

Роли (admin, developer, auditor) реализованы через:

1. **API-уровень**: Проверка разрешений при каждом вызове
2. **Фильтрация возможностей**: Разные наборы доступных операций для каждой роли
3. **UI адаптация**: Интерфейс отображает только доступные действия для текущей роли

### Реакция на нарушения Guardrails

При попытке выполнить запрещенное действие:

1. **Блокировка**: Действие немедленно блокируется
2. **Логирование**: Создается запись в аудит-логе
3. **Уведомление**: Отправка уведомления соответствующему администратору или аудитору
4. **Альтернативное предложение**: Предложение безопасного способа выполнения задачи

## 3. Структура аудиторского лога

### Содержание аудит-лога

Каждая запись в аудит-логе содержит:

1. **Идентификация**: Кто инициировал действие (роль, пользователь)
2. **Контекст**: Тип задачи, проект, файлы, затронутые ресурсы
3. **AI-модель**: Какая модель была использована
4. **Параметры**: Переданные параметры и настройки
5. **Результат**: Успех, ошибка, заблокировано
6. **Временные метки**: Время начала и завершения
7. **Сессия**: ID сессии для трассировки

## 4. Динамика RAG и ChromaDB

### Частота переиндексации

Процесс переиндексации:

1. **Автоматическая**: При обнаружении изменений в файлах проекта
2. **По расписанию**: Периодическая полная переиндексация (например, раз в день)
3. **По команде**: При необходимости полного обновления индекса

### Рекомендации для индексации

Агент может рекомендовать документы для индексации на основе:

1. **Анализа частоты обращений**: Часто запрашиваемые документы
2. **Связи с текущими задачами**: Документы, связанные с выполняемыми задачами
3. **Архитектурной важности**: Ключевые архитектурные документы и ADR

## 5. "Восстановление" как процесс

### Компоненты, требующие восстановления

Помимо Knowledge Graph, требуют восстановления:

1. **ИИ-планирование**: Логика планирования задач и оптимизация
2. **Fallback-механизмы**: Резервные системы при недоступности основных компонентов
3. **Интеграции**: Некоторые внешние интеграции могут быть частично работоспособны
4. **Тесты**: Некоторые тесты могут требовать обновления для соответствия текущему состоянию

### Последующие шаги после восстановления

После завершения "восстановления" планируется:

1. **Переход к стабильной версии**: Улучшение надежности и производительности
2. **Расширение функциональности**: Добавление новых возможностей
3. **Улучшение масштабируемости**: Поддержка больших проектов и команд
4. **Интеграция с новыми инструментами**: Расширение экосистемы интеграций
