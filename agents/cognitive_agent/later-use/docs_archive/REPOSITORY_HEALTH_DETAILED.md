# Подробное объяснение мониторинга здоровья репозитория

## 1. Git-анализ и система самовосстановления

### Статус реализации Git-анализа

На данный момент реализованы следующие компоненты Git-анализа:

1. **scan_git_diff()** - для обнаружения изменений в файлах
   - Использует `git status` и `git diff --name-only HEAD^ HEAD` для определения измененных файлов
   - Сравнивает хэши файлов до и после изменений
   - Определяет тип изменений (новые, измененные, удаленные файлы)

2. **extract_decision_markers()** - для анализа истории коммитов
   - Анализирует количество коммитов и выявляет решения в истории
   - Извлекает ключевые слова и маркеры из сообщений коммитов
   - Позволяет отслеживать активность разработки

### Дополнительная интеграция

Для полного анализа коммитов, вызывающих конфликты или падение CI, требуется:

1. **Интеграция с CI/CD системами**:
   - Доступ к логам CI/CD (GitHub Actions, GitLab CI, Jenkins)
   - Сопоставление времени падений тестов с временем коммитов
   - Анализ причин падений через интеграцию с Job Automation Agent

2. **Расширенный анализ Git-истории**:
   - Поиск строк конфликтов в логах слияния
   - Анализ сообщений об ошибках при `git merge` и `git rebase`
   - Идентификация коммитов, вызывающих regressions

### Определение конфликтных коммитов

Агент может определять конфликтные коммиты следующими способами:

1. **Анализ логов слияния**: Поиск строк вроде "CONFLICT", "merge conflict", "Automatic merge failed"
2. **Сопоставление с падениями тестов**: Сравнение времени коммита с временем падения тестов
3. **Анализ изменений в критических файлах**: Выявление изменений в файлах, связанных с упавшими тестами

## 2. Глубина анализа CI/CD падений

### Текущая глубина анализа

Агент может:

1. **Получать логи ошибок из CI** через Job Automation Agent
2. **Сопоставлять ошибки с недавними изменениями кода**
3. **Анализировать stack trace** и определять, какие файлы и функции связаны с ошибкой

### Пример интеллектуального анализа

При падении теста `<test_user_creation>` агент может:

1. **Анализировать stack trace** для определения файла и функции, вызвавшей ошибку
2. **Сканировать Git-историю** на предмет изменений в связанном файле
3. **Выдавать рекомендацию** вроде: "Вероятно, ошибка связана с изменением в файле user_model.py коммитом от [дата] пользователя [имя]"

### Механизм анализа

```python
def analyze_ci_failure(self, failure_log: str):
    # 1. Извлечь имя упавшего теста и stack trace
    test_name = self.extract_test_name(failure_log)
    affected_files = self.extract_affected_files(failure_log)

    # 2. Найти недавние коммиты, изменяющие эти файлы
    recent_changes = self.scan_recent_commits(affected_files)

    # 3. Сопоставить причину ошибки с изменениями
    correlation = self.correlate_error_with_changes(failure_log, recent_changes)

    # 4. Сгенерировать рекомендацию
    return {
        "likely_cause": correlation,
        "recent_commits": recent_changes,
        "recommended_fix": self.generate_fix_suggestion(correlation)
    }
```

## 3. Управление качеством покрытия кода

### Настройка порогов покрытия

На данный момент:

- **Порог покрытия 50%** установлен в конфигурации pytest как минимальный
- Этот порог **не настраивается динамически** через конфигурационные файлы
- Порог жестко задан в конфигурации тестирования

### Определение плохо покрытых участков

Агент определяет плохо покрытые участки через:

1. **Анализ отчетов о покрытии** (например, через `coverage.xml` или `lcov.info`)
2. **Проверку каждого файла и функции** отдельно
3. **Выявление непокрытых строк кода** в отчетах покрытия
4. **Сопоставление с важностью функций** (если доступна информация об этом)

### Потенциальная настраиваемость

Для настройки разных порогов покрытия для разных модулей потребуется:

1. **Расширение конфигурации** для поддержки порогов по модулям
2. **Интеграция с системой важности модулей** (например, через IT-Compass маркеры)
3. **Обновление системы рекомендаций** для учета различных порогов

## 4. Самовосстановление агента vs. самовосстановление репозитория

### Текущая архитектура самовосстановления

SelfHealingSystem в enterprise-версии фокусируется на:

- **Внутреннем состоянии агента** (время отклика, уровень ошибок, использование ресурсов)
- **Обнаружении аномалий в работе самого агента**
- **Автоматическом восстановлении после сбоев агента**

### Возможности по "самовосстановлению" репозитория

При обнаружении множественных падений тестов после коммита X:

1. **Агент генерирует рекомендацию** об откате или исправлении коммита
2. **Создает задачу** (через Job Automation Agent или систему тикетов)
3. **Предлагает решение**, но не выполняет автоматический откат без подтверждения

### Будущие возможности

Планируется реализовать:

1. **Создание автоматических PR** с исправлениями
2. **Интеллектуальный откат** проблемных коммитов с последующей проверкой
3. **Интеграция с Git** для автоматических операций восстановления
