    Ответ: На Windows PowerShell используется .venv\Scripts\Activate.ps1, а не source.

    2. Что такое виртуальное окружение

    Вопрос: Расскажи что такое виртуальное окружение

    Ответ: - Изолированное пространство для проекта с собственными зависимостями - Решает конфликты версий между проектами - Пример: FastAPI 0.10.0 в Проекте A, FastAPI 0.136.1 в Проекте B - Команды: python -m venv .venv, .venv\Scripts\Activate.ps1, deactivate

    3. Скрипт проверки виртуального окружения

    Вопрос: Как именно проверяется виртуальное окружение — последовательно с подтверждающими фактами

    Ответ: Создан чеклист из 6 шагов: 1. Активация (промпт показывает (.venv)) 2. python --version и where python (путь внутри .venv) 3. sys.prefix (должен быть .venv) 4. pip list (есть fastapi, pydantic, langchain) 5. pip show fastapi (Location в .venv\Lib\site-packages) 6. Тестовый импорт (без ошибок)

    Создан файл: docs/CHECKLIST-VENV-WINDOWS.md

    4. Дубликаты файлов (2), _REAL

    Вопрос: Что будет если два файла будут называться README.md и README (2).md?

    Ответ: - Путаница для HR, рекрутеров, технических лидов - GitHub показывает только README.md на главной - Потеря истории изменений - Рекомендация: Оставить README.md, удалить дубликаты или переместить в docs/archive/

    6. Методика принятия решений: что удалить/сохранить

    Вопрос: Как ты будешь решать что удалить — пункт 1? Без выполнения, просто ответь.

    Ответ: Алгоритм: 1. Сравнение по дате модификации (позднее = актуальнее) 2. Сравнение по размеру (разница >20% → проверить содержимое) 3. Проверка через Git-историю (файл без истории → кандидат на удаление) 4. Сравнение содержимого (diff) 5. Для конфигов — ручная проверка ключевых секций 6. Файлы-мусор (.vsix, %USERPROFILE%.gitconfig) — удалить без раздумий

    9. Что происходит при git push

    Вопрос: Что происходит при git push?

    Ответ: Процесс: 1. Git проверяет доступ к origin и актуальность ветки 2. Отправляет коммиты, деревья файлов, блобы, теги 3. На GitHub обновляется refs/heads/main, запускаются GitHub Actions 4. Все становятся видны команде

    В вашем случае: 102 локальных коммита будут отправлены, запустятся CI/CD workflows, Dependabot может создать новые PR.

    10. Проверка конфликтов перед пушем

    Вопрос: Как именно проверяют на конфликты перед пушем?

    Ответ: Чеклист: 1. git fetch origin 2. git status (впереди или позади remote) 3. git log --oneline origin/main..HEAD (просмотр коммитов) 4. git diff origin/main..HEAD (различия) 5. git rebase --dry-run origin/main (проверка конфликтов) 6. make test (локальные тесты перед пушем) 7. git push origin main

    14. Контейнеризация

    Вопрос: Контейнеризированы это значит каждый можно установить независимо от другого и всё для установки готово?

    Ответ: Нет. Контейнеризация означает: - Упаковка сервиса с зависимостями в Docker-контейнер - Изоляция от других сервисов - Переносимость (работает везде с Docker)

    НО: Сервисы зависят друг от друга (API, БД, очереди), поэтому нужна оркестрация (Docker Compose, Kubernetes).

    Запуск: docker-compose up (запускает ВСЕ сервисы сразу).

    15. Почему 14 микросервисов

    Вопрос: Почему автор сделал не один, а 14 микросервисов?

    Ответ: Практические причины: - Разделение ответственности (каждый сервис — одна задача) - Независимое масштабирование - Отказоустойчивость (падение одного не ломает все) - Разные технологии (Python, Node.js, Electron)

    Демонстрационные причины (для портфолио): - Каждый сервис — доказательство компетенции - 1595 маркеров IT-компетенций (IT-Compass) - Демонстрация системного мышления и оркестрации ИИ
