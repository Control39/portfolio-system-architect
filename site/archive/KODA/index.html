    KODA.md — Контекст для взаимодействия с проектом

    Дата создания: 22 апреля 2026 г. Тип проекта: Продукт с кодом — когнитивная архитектура и система автоматизации

    📋 Обзор проекта

    Назначение

    Portfolio System Architect — это производственно-готовая экосистема когнитивной архитектуры, демонстрирующая трансформацию от полного отсутствия IT-опыта до создания 14 интегрированных микросервисов за 2 года. Проект представляет собой:

    1. Cognitive Automation Agent (CAA) — автономная система автоматизации с интеллектуальными возможностями
    2. IT-Compass — методология объективного измерения компетенций через 83 проверочных маркера в 19 IT-доменах
    3. Систему оркестрации ИИ — использование ИИ как исполнительного слоя под строгим архитектурным контролем

    Ключевые технологии

    Категория Технологии
    Язык программирования Python 3.10+
    Фреймворки FastAPI, LangChain, Streamlit
    База данных PostgreSQL 16, ChromaDB (векторная БД), Redis 7
    Контейнеризация Docker, Docker Compose
    Оркестрация Kubernetes (Kustomize), GitOps
    Мониторинг Prometheus, Grafana, AlertManager
    API Gateway Traefik v3.0
    CI/CD GitHub Actions
    Безопасность Trivy, Bandit, Sealed Secrets
    ИИ/ML LangChain, ChromaDB, Sentence Transformers

    Основные компоненты

    1. Cognitive Automation Agent (apps/cognitive-agent/)

    Автономная система для управления проектами с интеллектуальными возможностями: - Project Scanner — интеллектуальное сканирование технологического стека - Task Planner — проактивное планирование задач - Learning System — система самообучения на основе метрик - Trigger System — система триггеров и автоматических действий

    2. IT-Compass (apps/it-compass/)

    Методологическое ядро для объективного измерения компетенций: - 83 проверочных маркера в 19 IT-доменах - Модульный Python-пакет с JSON-схемой - Streamlit UI для визуализации - Интеграция с портфолио и облачными решениями - Психологическая поддержка (профилактика выгорания)

    3. Microservices Architecture

    14 интегрированных микросервисов, 12 из которых контейнеризованы: - Cloud Reason Engine — принятие решений на основе ИИ с объяснимой логикой - Portfolio Organizer — автоматический сбор доказательств и картирование компетенций - RAG API — векторный поиск и документная интеллектуализация - System Proof — автоматическая валидация критериев производственной готовности - ML Model Registry — версионированные ML-модели с A/B тестированием - API Gateway — единая точка входа с безопасностью и наблюдаемостью

    🚀 Сборка и запуск

    Предварительные требования

    • Docker & Docker Compose (Docker Desktop или Docker Engine)
    • Git с Git LFS (Large File Storage)
    • Python 3.10+ (для разработки)
    • 4GB+ RAM для контейнеров

    Установка зависимостей

    # Создание виртуального окружения (если отсутствует)
make install

# Или вручную:
python -m venv .venv
# Windows:
.venv\Scripts\activate
# Linux/Mac:
source .venv/bin/activate

# Установка зависимостей
pip install -r requirements-dev.txt
pip install -e .

# Установка pre-commit hooks
pre-commit install

    Запуск среды разработки

    # Запуск всех сервисов через Docker Compose
make dev

# Или вручную:
docker-compose -f docker-compose.yml -f docker-compose.monitoring.yml up -d

# Проверка состояния сервисов
docker-compose ps

    🛠️ Правила разработки

    Стиль кодирования

    1. Python-стандарты:
    2. Использование Black для форматирования
    3. isort для сортировки импортов
    4. Ruff для линтинга (заменяет flake8, pylint)

    5. MyPy для статической типизации

    6. Структура кода:

    7. Файлы тестов: test_*.py
    8. Функции тестов: test_*
    9. Классы тестов: Test*

    10. Пути тестов: apps/, tests/, tests/e2e/, tests/unit/

    11. Конфигурация линтеров:

    12. pyrightconfig.json — настройки статической типизации
    13. .ruff_cache/ — кэш линтера Ruff
    14. .pre-commit-config.yaml — конфигурация pre-commit хуков

    Практики тестирования

    1. Уровни тестирования:
    2. Юнит-тесты — изолированные тесты отдельных компонентов
    3. Интеграционные тесты — тесты взаимодействия компонентов
    4. E2E тесты — полные сценарии использования (помечены как slow)

    5. Нагрузочные тесты — с помощью Locust

    6. Покрытие кода:

    7. Минимальный порог: 95% (настроен в pyproject.toml)
    8. Отчёты: HTML + терминал с пропусками

    9. Пути покрытия: apps/, src/

    10. Маркеры тестов:

    11. @pytest.mark.e2e — E2E тесты (запускаются отдельно)

    12. @pytest.mark.slow — медленные тесты

    13. Файлы тестов:

    14. Расположение: tests/, apps/*/tests/
    15. Имя: test_*.py
    16. Импорты: тестовые утилиты из tests/conftest.py

    Правила контрибуции

    1. Git workflow:
    2. Использовать ветки с понятными именами
    3. Коммиты с сообщениями на английском
    4. Protected branches для main

    5. Code review для всех PR

    6. Безопасность:

    7. Не коммитить секреты (использовать .secrets.baseline)
    8. Регулярные Trivy-сканы
    9. Bandit для Python-безопасности

    10. Sealed Secrets для Kubernetes

    11. Документация:

    12. Все архитектурные решения в docs/architecture/decisions/ (ADR)
    13. Обновление README при изменении API

    14. Чек-листы для сложных изменений

    15. CI/CD:

    16. Все изменения должны проходить CI
    17. Автоматические тесты при push
    18. GitOps-подход для деплоя

    Архитектурные принципы

    2. apps/cognitive-agent/ — автономное выполнение (уровень реализации)
    3. codeassistant/ — аналитические skills (уровень анализа)

    4. Чёткие границы между микросервисами

    5. Contract-first design:

    6. API-контракты определяются заранее
    7. Валидация через Pydantic

    8. OpenAPI/Swagger документация

    9. Observability by default:

    10. Метрики для всех сервисов
    11. Логирование в структурированном формате

    12. Health checks для всех контейнеров

    13. Security by design:

    14. JWT аутентификация
    15. Rate limiting через Traefik
    16. Network policies в Kubernetes
    17. Регулярные security сканы

    Рекомендации для анализа

    1. При чтении кода:
    2. Обращать внимание на ADR в docs/architecture/decisions/
    3. Проверять соответствие архитектурным принципам

    4. Искать интеграционные паттерны между микросервисами

    5. При генерации кода:

    6. Следовать стилю Black + isort + Ruff
    7. Добавлять type hints
    8. Писать тесты для нового кода

    9. Использовать Pydantic для валидации

    10. При анализе проблем:

    11. Проверять health checks сервисов
    12. Анализировать логи через docker-compose logs
    13. Использовать Grafana для метрик
    14. Проверять security сканы

    Критические пути

    1. Запуск среды: make dev → проверить все сервисы
    2. Запуск тестов: make test → покрытие > 95%
    3. Линтинг: make lint → без ошибок
    4. Деплой: deployment/k8s/ → GitOps workflow
