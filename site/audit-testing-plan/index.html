    План тестирования и валидации инструмента repo-audit

    Цели тестирования

    1. Убедиться, что инструмент корректно проверяет все пункты чек-листа.
    2. Проверить интеграцию с GitHub Actions.
    3. Валидировать работу AI-навыка.
    4. Оценить производительность и устойчивость к ошибкам.

    Типы тестов

    1. Unit-тесты

    Покрытие модулей repo_audit/checks/*, repo_audit/checker.py, repo_audit/report.py.

    Пример:

    def test_required_files_check():
    check = RequiredFilesCheck()
    with tempfile.TemporaryDirectory() as tmp:
        # Создаём только README.md
        open(os.path.join(tmp, 'README.md'), 'w').close()
        result = check.run(AuditContext(repo_root=tmp))
        assert not result.passed
        assert 'LICENSE' in result.message

    2. Интеграционные тесты

    Запуск инструмента на тестовых репозиториях с разной структурой.

    Тестовые репозитории: - test_repos/perfect/ — соответствует всем уровням. - test_repos/empty/ — пустой репозиторий. - test_repos/mixed/ — частичное соответствие.

    3. End-to-end тесты

    Полный цикл: установка инструмента, запуск аудита, генерация отчёта.

    Сценарий:

    cd test_repos/perfect
python -m repo_audit run --level all --format json
# Проверяем, что выходной JSON содержит ожидаемые результаты.

    4. Тестирование GitHub Actions

    Используем act для локального запуска workflow.

    act -W .github/workflows/repo-audit.yml -e test-event.json

    5. Тестирование AI-навыка

    • Мокаем вызов инструмента repo_audit.
    • Проверяем, что навык корректно парсит отчёт и генерирует рекомендации.
    • Тестируем обработку ошибок (например, отсутствие конфига).

    Критерии приёмки

    Критерий Метод проверки Ожидаемый результат
    Все обязательные проверки уровня 1 работают Unit-тесты 100% прохождение
    Инструмент запускается без ошибок на текущем репозитории E2E тест Exit code 0, отчёт создан
    Workflow проходит на GitHub Actions Запуск в реальном репо Зелёный статус
    AI-навык отвечает на запросы Интеграционный тест с SourceCraft Ответ содержит рекомендации
    Производительность: аудит репо < 30 сек Замер времени Уложиться в лимит

    Инструменты тестирования

    • pytest — для unit и интеграционных тестов.
    • coverage — для измерения покрытия кода (цель ≥80%).
    • act — для локального запуска GitHub Actions.
    • tox — для тестирования на разных версиях Python (3.9, 3.10, 3.11).
    • hypothesis — для property-based тестирования (опционально).

    План выполнения

    Этап 1: Базовые unit-тесты

    • Написать тесты для 10 самых важных проверок.
    • Добиться покрытия >70%.

    Этап 2: Интеграционные тесты

    • Создать тестовые репозитории в tests/fixtures/.
    • Написать скрипт, который запускает аудит и сравнивает с эталонным выводом.

    Этап 3: E2E тестирование

    • Добавить pytest-фикстуру, которая устанавливает инструмент в виртуальное окружение.
    • Проверить работу CLI с разными аргументами.

    Этап 4: Тестирование workflow

    • Создать test-event.json для эмуляции события PR.
    • Запустить act и убедиться, что шаги выполняются без ошибок.

    Этап 5: Валидация AI-навыка

    • Написать mock-сервер, имитирующий SourceCraft API.
    • Проверить, что навык корректно обрабатывает входящие события.

    Отчёт о тестировании

    После выполнения тестов будет сгенерирован отчёт в формате JUnit XML (для интеграции с CI) и HTML-отчёт покрытия.

    Следующие шаги

    1. Переключиться в режим Code для реализации тестов.
    2. Настроить pytest и coverage в pyproject.toml.
    3. Добавить GitHub Actions workflow для запуска тестов при каждом PR.
    4. Запустить тесты на текущем репозитории и исправить выявленные проблемы.
